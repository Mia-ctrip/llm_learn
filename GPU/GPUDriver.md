# NVIDIA GPU 驱动内核模块完全指南

## 前言

NVIDIA 的 Linux 驱动由**内核模块**和**用户态组件**两部分组成。2022 年起，NVIDIA 推出了开源内核模块，与传统的专有内核模块形成两条并行的技术路线。本文档系统梳理两者的区别、适用范围，以及实际部署中的选择策略。

> 💡 **重要说明**：所谓"开源"目前仅限于**内核模块**部分，用户态的 CUDA、OpenGL、Vulkan 等核心组件**仍然闭源**。这与 AMD 的 AMDGPU（完全开源）在开放程度上仍有差距。

---

## 一、两种内核模块概览

### 1.1 专有内核模块（Proprietary）

- **闭源**，以二进制形式分发
- 长期以来是 Linux 下使用 NVIDIA GPU 的默认方式
- 支持从 Kepler 到 Hopper 的几乎所有架构
- 功能最完整，所有特性（CUDA、vGPU、图形等）均可用
- 缺点：内核被标记为 "tainted"，不符合部分发行版的开源策略

### 1.2 开源内核模块（Open Kernel Modules）

- 2022 年 5 月（R515）首次发布，采用 **MIT / GPLv2 双许可证**
- 自 R560（2024 年 7 月）起，成为 Turing 及更新架构的**官方推荐**
- 可与 Linux 内核更好地集成，发行版可直接打包分发

### 1.3 开源模块为什么可行？—— GSP 固件

开源模块的关键前提是 GPU 上有 **GSP（GPU System Processor）** 协处理器：

```
┌─────────────────────────┐
│  用户态（仍闭源）         │  CUDA、OpenGL、Vulkan 驱动
├─────────────────────────┤
│  开源内核模块            │  较薄的桥接层
├─────────────────────────┤
│  GSP 固件（闭源 blob）   │  真正的硬件管理逻辑
├─────────────────────────┤
│       GPU 硬件           │
└─────────────────────────┘
```

NVIDIA 将大部分硬件管理逻辑转移到 GSP 固件中运行，内核模块只作为一个薄桥接层，因此才有条件开源。**没有 GSP 的老架构（Maxwell/Pascal/Volta）硬件上就无法使用开源模块。**

## 二、架构与模块支持对照表

下表中"✅ 可用"表示该架构**可以**使用对应模块，"❌ 不可用"表示受硬件或官方策略限制**无法**使用。

| 架构 | 发布年份 | 数据中心型号 | 消费级 / 工作站 | 中国特供版 | 开源模块 | 专有模块 |
|------|---------|-------------|----------------|-----------|---------|---------|
| **Maxwell** | 2014 | Tesla M40 / M60 | GTX 900 系 | — | ❌ | ✅ |
| **Pascal** | 2016 | Tesla P4 / P40 / P100 | GTX 10 系、Titan Xp | — | ❌ | ✅ |
| **Volta** | 2017 | Tesla V100 / V100S | Titan V | — | ❌ | ✅ |
| **Turing** | 2018 | Tesla T4 | RTX 20 系、Quadro RTX | — | ✅ | ✅ |
| **Ampere** | 2020 | A100 / A40 / A30 / A10 / A2 | RTX 30 系、RTX A 系列 | A800 | ✅ | ✅ |
| **Ada Lovelace** | 2022 | L40 / L40S / L4 | RTX 40 系、RTX 6000/5000 Ada | **L20 / L2 / RTX 5880 Ada** | ✅ | ✅ |
| **Hopper** | 2022 | H100 / H200 | — | **H20 / H800** | ✅ | ✅ |
| **Blackwell** | 2024-2025 | B100 / B200 / GB200 | RTX 50 系、RTX PRO Blackwell 系列 | — | ✅ | ❌ |

### 关键判断口诀

> **有 GSP → 能开源；没 GSP → 只能专有**
>
> - **Turing 起有 GSP**（T4、A100、L20、H20、RTX 5880 Ada…）
> - **Blackwell 起砍掉专有**（B100、B200、RTX 50 系、RTX PRO Blackwell…）

### 中国特供卡的驱动特性

H20、H800、L20、L2、RTX 5880 Ada 等特供版本只是在算力/带宽上被限制，**驱动层面与同架构的原版完全一致**，可以按照对应架构的规则选择模块。

## 三、常见命名误区

NVIDIA 的型号命名容易混淆，以下几个典型例子：

| 型号 | 架构 | 产品线 | 说明 |
|------|------|--------|------|
| **GeForce RTX 5090 / 5080** | Blackwell | 消费级（游戏） | 所谓"RTX 50 系" |
| **RTX PRO 5000 Blackwell** | Blackwell | 专业工作站 | 2025 年新命名，带 "PRO" |
| **RTX 5000 Ada** | Ada Lovelace | 专业工作站 | 上一代工作站卡 |
| **RTX 5880 Ada** | Ada Lovelace | 专业工作站（中国特供） | **不是 Blackwell！** |
| **RTX A5000** | Ampere | 专业工作站 | 更老一代 |

**判断方法**：看型号后缀
- "**Blackwell**" → Blackwell 架构，只能用开源模块
- "**Ada**" → Ada Lovelace 架构，两种模块都可用
- "**A**xxxx" → Ampere 架构

## 四、内核模块选择的决策因素

**"用什么模块"不只是看架构**，实际决策需要综合考虑多个维度：

```
1. GPU 架构            ← 决定"能不能"（硬件约束）
2. 驱动版本            ← 决定"版本够不够"
3. 同机混插情况         ← 决定"冲不冲突"
4. 发行版 / 内核版本    ← 决定"好不好装"
5. 业务功能需求         ← 决定"够不够用"
```

### 4.1 驱动版本门槛

| 驱动分支 | 开源模块状态 |
|---------|-------------|
| R470 及更早 | ❌ 无开源模块 |
| R515 ~ R535 | ⚠️ 早期版本，不推荐生产 |
| R550 | ✅ 稳定 |
| **R560**（2024.07） | ✅ **开源成为官方默认推荐** |
| R570+ | ✅ 必须（Blackwell 支持） |

### 4.2 同机混插约束 ⚠️

> **同一台机器上的所有 NVIDIA GPU 必须使用同一种内核模块**

| 混插组合 | 能用的模块 |
|---------|-----------|
| V100 + A100 | 只能专有（V100 不支持开源） |
| T4 + A100 + H100 | 两种都可以 |
| H100 + B200 | 只能开源（B200 不支持专有） |
| V100 + B200 | ❌ **无法共存** |

这对老集群升级尤其关键：从 V100 逐步替换到 H100 的混插过渡期，必须继续使用专有模块。

### 4.3 业务功能差异

| 功能 | 开源 | 专有 |
|------|------|------|
| CUDA、NVENC/NVDEC | ✅ | ✅ |
| MIG（A100/H100 切分） | ✅ | ✅ |
| GPUDirect RDMA | ✅ | ✅ |
| **vGPU（虚拟化）** | ⚠️ 支持较晚 | ✅ |

如业务涉及 **vGPU 虚拟化**，建议继续使用专有模块。

## 五、部署场景推荐

| 场景 | GPU 型号示例 | 推荐模块 | 备注 |
|------|-------------|---------|------|
| 老旧推理集群 | Tesla V100 / P40 | **专有** | 硬件无 GSP，无选择 |
| 老集群升级过渡期 | V100 + A100 混插 | **专有** | 整机一致性约束 |
| 新 AI 推理部署 | T4 / L4 / L20 / L40S | **开源** | R560+ |
| 新 AI 训练集群 | A100 / A800 / H100 / H20 | **开源** | R560+ 官方推荐 |
| AI 工作站 | RTX 5880 Ada / RTX 6000 Ada | **开源** | R550+ |
| 下一代部署 | B100 / B200 / RTX PRO Blackwell | **开源**（唯一选择） | R570+，CUDA 12.8+ |
| vGPU 虚拟化场景 | 任意支持型号 | 专有 | 功能完整度更高 |

## 六、实用命令

### 查看 GPU 型号和架构
```bash
nvidia-smi --query-gpu=name,compute_cap --format=csv
```

**计算能力（Compute Capability）对照**：
| 数值 | 架构 |
|------|------|
| 7.0 | Volta |
| 7.5 | Turing |
| 8.0 / 8.6 | Ampere |
| 8.9 | Ada Lovelace |
| 9.0 | Hopper |
| 10.0 / 12.0 | Blackwell |

### 查看当前加载的内核模块类型
```bash
modinfo nvidia | grep license
# license: Dual MIT/GPL  → 开源模块
# license: NVIDIA        → 专有模块
```

### 查看 GSP 固件状态
```bash
nvidia-smi -q | grep -i gsp
```

### 安装时选择模块类型
```bash
# 使用 .run 安装程序
sudo ./NVIDIA-Linux-x86_64-xxx.run --kernel-module-type=open         # 开源
sudo ./NVIDIA-Linux-x86_64-xxx.run --kernel-module-type=proprietary  # 专有

# Ubuntu APT 包名区分
sudo apt install nvidia-driver-550-open    # 开源
sudo apt install nvidia-driver-550         # 专有
```

## 七、部署前检查清单

在为一批 GPU 选择驱动方案前，先回答以下四个问题：

1. ✅ **这批 GPU 里最老的是什么架构？**（决定整机下限）
2. ✅ **业务软件栈要求的 CUDA 最低版本？**（决定驱动版本下限）
3. ✅ **是否需要 vGPU 或特殊虚拟化？**（可能强制专有）
4. ✅ **后续是否会混插新老架构？**（影响长期策略）

把这四个问题答清楚，模块选择的答案就自然确定了。

## 八、总结

| 结论 | 说明 |
|------|------|
| **架构是底层硬约束** | Maxwell/Pascal/Volta 只能专有；Blackwell 只能开源 |
| **中间地带有选择权** | Turing ~ Hopper 两种都可用，默认推荐开源 |
| **整机必须一致** | 同台服务器所有 GPU 必须使用同一种模块 |
| **开源是未来方向** | NVIDIA 新架构已全面转向开源模块 |
| **用户态仍然闭源** | CUDA、图形
