# LLM 推理速度学习指南

## 1. 推理 vs 训练

### 1.1 核心区别

**推理（Inference）**：
- ✅ 只有**前向传播**（Forward Pass）
- ✅ 计算神经元输出
- ✅ 计算损失值（可选）
- ❌ **没有反向传播**（Backward Pass）
- ❌ **没有梯度计算**
- ❌ **没有参数更新**

**训练（Training）**：
- ✅ 前向传播 + 反向传播
- ✅ 梯度计算和参数更新
- ✅ 需要存储中间激活值
- ⚠️ 显存需求是推理的 3-6 倍
  - 模型权重（1x）：和推理相同 ✓
  - **梯度**（+1x）：每个参数的导数
  - **优化器状态**（+2x）：Adam 需要 2 个动量
  - **激活值**（+0.5-3x）：所有层的输出（反向传播需要）

### 1.2 为什么推理更快？

```
训练时间 = 前向传播 + 反向传播 + 参数更新
推理时间 = 前向传播

理论上：推理速度 ≈ 训练速度的 2-3 倍
```

---

## 2. 影响推理速度的核心因素

### 2.1 模型规模（参数量 + 层数）

**计算量公式**：
```
FLOPs = 2 × 参数量 × 序列长度
```

**影响**：
- 35B 模型比 7B 模型慢约 **5 倍**
- 层数越多，**串行计算**越多，延迟越高

**示例**：
```
Qwen2.5-7B:  2 × 7B × 2000 tokens = 28 TFLOPs
Qwen3.5-35B: 2 × 35B × 2000 tokens = 140 TFLOPs
→ 35B 模型需要的计算量是 7B 的 5 倍
```

---

### 2.2 输入长度（序列长度）

对 Transformer 模型，**注意力机制的复杂度**：

```
计算复杂度 = O(n²)
```

**影响**：
- 序列长度翻倍 → 注意力计算量翻 **4 倍**
- KV Cache 线性增长

**示例**：
```
序列长度 1K tokens：计算量 = O(1000²) = 1M 次运算
序列长度 2K tokens：计算量 = O(2000²) = 4M 次运算 (4倍)
序列长度 4K tokens：计算量 = O(4000²) = 16M 次运算 (16倍)
```

---

### 2.3 显卡算力（Compute Power）

**衡量指标：TFLOPS**（每秒万亿次浮点运算）

| GPU | FP16 算力 | INT8 算力 | INT4 算力 |
|-----|----------|----------|----------|
| **RTX 4090** | 82 TFLOPS | ~165 TFLOPS | ~330 TFLOPS |
| **A100** | 312 TFLOPS | 624 TFLOPS | - |
| **H100** | 989 TFLOPS | 1979 TFLOPS | ~3958 TFLOPS |

**公式**：
```
理论推理时间 = 计算量 (FLOPs) / GPU算力 (TFLOPS)
```

**示例**：
```
Qwen2.5-7B, 2K tokens, RTX 4090:
理论时间 = 28 TFLOPs / 82 TFLOPS ≈ 0.34 秒（仅前向传播）
```

---

### 2.4 内存带宽（Memory Bandwidth）⚠️ 最大瓶颈

#### 为什么内存带宽是瓶颈？

**关键事实**：
```
每生成一个 token，都要读取一遍完整的模型参数！
```

**原因**：
- 自回归生成：逐个 token 生成
- 每个 token 的生成都需要全部权重参与计算
- **读取数据的时间 > 计算时间**

#### 内存带宽速查表

| GPU | 内存带宽 | INT4 35B 理论最大速度 |
|-----|---------|---------------------|
| **RTX 4090** | 1008 GB/s | 1008 / 18 ≈ **56 tokens/秒** |
| **A100** | 1935 GB/s | 1935 / 18 ≈ **107 tokens/秒** |
| **H100** | 3352 GB/s | 3352 / 18 ≈ **186 tokens/秒** |

**公式**：
```
理论最大速度 (tokens/秒) = 内存带宽 (GB/s) / 模型大小 (GB)
```

#### 实际案例：Qwen3.5-35B INT4

**模型大小**：
```
35B × 0.5 bytes (INT4) = 17.5 GB ≈ 18 GB
```

**RTX 4090 推理速度**：
```
理论最大速度 = 1008 GB/s / 18 GB = 56 tokens/秒
实际速度 ≈ 30-40 tokens/秒（考虑 KV Cache 和开销）
```

**为什么实际速度更慢？**
1. KV Cache 占用带宽（随序列变长而增加）
2. 中间激活值的读写
3. 显存碎片和调度开销

---

### 2.5 批处理大小（Batch Size）

**吞吐量 vs 延迟**：

| Batch Size | 吞吐量 | 单请求延迟 | GPU 利用率 | 适用场景 |
|-----------|--------|----------|-----------|---------|
| 1 | 低 | **极低** | ~30% | 实时对话 |
| 8 | 中 | 中 | ~70% | API 服务 |
| 32 | **高** | 高 | ~95% | 批量处理 |

**关系**：
```
总吞吐量 (tokens/秒) ≈ 单batch速度 × Batch_size × 0.8

例子：
Batch=1:  40 tokens/秒
Batch=8:  40 × 8 × 0.8 = 256 tokens/秒（但每个请求要等8倍时间）
```

---

### 2.6 量化精度（Quantization）

量化通过降低权重精度来提升速度：

| 精度 | 模型大小 | 相对速度 | 内存带宽压力 | 精度损失 |
|-----|---------|---------|------------|---------|
| **FP32** | 4 字节/参数 | 1x | 最高 | 无 |
| **FP16** | 2 字节/参数 | **2x** | 50% | 极小 |
| **INT8** | 1 字节/参数 | **3-4x** | 25% | 小 |
| **INT4** | 0.5 字节/参数 | **4-6x** | 12.5% | 中等 |

**速度提升原因**：
1. **减少数据传输**：模型更小 → 内存带宽压力减半
2. **更快的计算**：INT 运算比浮点快
3. **更高的并行度**：可以同时处理更多数据

**示例**：35B 模型
```
FP16:  35 × 2 = 70 GB  → 速度 = 1008/70 = 14 tokens/秒
INT8:  35 × 1 = 35 GB  → 速度 = 1008/35 = 29 tokens/秒 (2倍)
INT4:  35 × 0.5 = 18 GB → 速度 = 1008/18 = 56 tokens/秒 (4倍)
```

---

## 3. 推理的两个阶段

### 3.1 Prefill 阶段（处理输入）

**做什么**：
- 一次性处理全部输入 tokens
- 计算所有输入的注意力
- 生成初始 KV Cache

**特点**：
- ✅ **计算密集型**（Compute-bound）
- ✅ 可以并行处理所有输入
- ✅ GPU 利用率高（接近 100%）

**速度公式**：
```
Prefill 时间 = (输入长度 × 参数量 × 2) / GPU算力
```

**示例**：
```
Qwen3.5-35B, 输入 1000 tokens, RTX 4090 (82 TFLOPS):
计算量 = 2 × 35B × 1000 = 70 TFLOPs
时间 = 70 / 82 ≈ 0.85 秒
```

---

### 3.2 Decode 阶段（生成输出）

**做什么**：
- 逐个生成新 token
- 每次只生成 1 个 token
- 不断更新 KV Cache

**特点**：
- ⚠️ **带宽密集型**（Memory-bound）
- ⚠️ 串行生成，无法并行
- ⚠️ GPU 利用率低（20-40%）

**速度公式**：
```
生成速度 (tokens/秒) = 内存带宽 (GB/s) / 模型大小 (GB)
```

**示例**：
```
Qwen3.5-35B INT4 (18 GB), RTX 4090 (1008 GB/s):
理论速度 = 1008 / 18 = 56 tokens/秒
实际速度 ≈ 30-40 tokens/秒
```

---

### 3.3 总推理时间

```
总时间 = Prefill 时间 + Decode 时间

= (输入长度 × 参数量 × 2) / GPU算力 
  + 输出长度 / (内存带宽 / 模型大小)
```

**示例**：生成一个 500 tokens 的回答
```
输入：1000 tokens
输出：500 tokens
模型：Qwen3.5-35B INT4 (18 GB)
硬件：RTX 4090 (82 TFLOPS, 1008 GB/s)

Prefill: 70 TFLOPs / 82 = 0.85 秒
Decode:  500 tokens / 40 tokens/秒 = 12.5 秒

总时间 ≈ 13.4 秒
```

---

## 4. 推理速度估算公式总结

### 4.1 快速估算法（Decode 为主）

**公式**：
```
生成速度 (tokens/秒) = 内存带宽 (GB/s) / 模型大小 (GB) × 效率系数

效率系数 ≈ 0.5 - 0.7（考虑 KV Cache 和开销）
```

**示例速查表**：

#### RTX 4090 (1008 GB/s)

| 模型 | 精度 | 模型大小 | 理论速度 | 实际速度 |
|-----|------|---------|---------|---------|
| Qwen2.5-7B | FP16 | 14 GB | 72 tokens/s | **50 tokens/s** |
| Qwen2.5-7B | INT4 | 3.5 GB | 288 tokens/s | **180 tokens/s** |
| Qwen3.5-35B | FP16 | 70 GB | 14 tokens/s | **10 tokens/s** |
| Qwen3.5-35B | INT4 | 18 GB | 56 tokens/s | **35 tokens/s** |

#### A100 (1935 GB/s)

| 模型 | 精度 | 模型大小 | 理论速度 | 实际速度 |
|-----|------|---------|---------|---------|
| Qwen2.5-7B | FP16 | 14 GB | 138 tokens/s | **100 tokens/s** |
| Qwen3.5-35B | INT4 | 18 GB | 107 tokens/s | **70 tokens/s** |

#### H100 (3352 GB/s)

| 模型 | 精度 | 模型大小 | 理论速度 | 实际速度 |
|-----|------|---------|---------|---------|
| Qwen2.5-7B | FP16 | 14 GB | 239 tokens/s | **180 tokens/s** |
| Qwen3.5-35B | INT4 | 18 GB | 186 tokens/s | **130 tokens/s** |

---

### 4.2 精确估算法

**完整公式**：
```
总时间 (秒) = Prefill时间 + Decode时间

Prefill时间 = (2 × 参数量 × 输入长度) / (GPU算力 × 10¹²)

Decode时间 = 输出长度 / (内存带宽 / (模型大小 × (1 + KV_Cache_Overhead)))

KV_Cache_Overhead ≈ 0.1 - 0.3（短序列到长序列）
```

## 5. 实战案例分析

### 案例 1：Qwen3.5-35B INT4 推理速度估算

**配置**：
- 模型：Qwen3.5-35B
- 量化：INT4 (0.5 bytes/参数)
- 硬件：RTX 4090
- 任务：生成一篇 500 字的文章

**步骤 1：计算模型大小**
```
模型大小 = 35B × 0.5 bytes = 17.5 GB ≈ 18 GB
```

**步骤 2：估算 Decode 速度**
```
理论速度 = 1008 GB/s / 18 GB = 56 tokens/秒
实际速度 = 56 × 0.65 (效率系数) ≈ 36 tokens/秒
```

**步骤 3：计算总时间**
```
输入：100 tokens (Prefill)
输出：500 tokens (Decode)

Prefill: 2 × 35B × 100 / (82 × 10¹²) ≈ 0.085 秒
Decode:  500 / 36 ≈ 14 秒

总时间 ≈ 14.1 秒
```

**结论**：
- ✅ 35B INT4 模型 + RTX 4090 = **可行**
- ✅ 生成速度：约 **36 tokens/秒**
- ✅ 用户体验：流畅（20+ tokens/秒即可）

---

### 案例 2：为什么长序列会变慢？

**场景**：同样的模型，处理不同长度的对话

| 对话轮数 | 上下文长度 | KV Cache | 实际速度 | 原因 |
|---------|-----------|---------|---------|-----|
| 第 1 轮 | 100 tokens | 0.2 GB | 36 tokens/s | KV Cache 小 |
| 第 5 轮 | 2000 tokens | 3.2 GB | 28 tokens/s | KV Cache 占用带宽 |
| 第 10 轮 | 8000 tokens | 12 GB | 18 tokens/s | KV Cache 接近模型大小 |

**结论**：
```
有效模型大小 = 权重 + KV Cache
实际速度 = 内存带宽 / 有效模型大小

长对话 → KV Cache 增大 → 有效模型大小增加 → 速度下降
```

---

### 案例 3：选择合适的硬件

**目标**：部署 Qwen3.5-35B，要求生成速度 ≥ 30 tokens/秒

**方案对比**：

| 方案 | 配置 | 模型大小 | 速度 | 成本 | 推荐度 |
|-----|------|---------|-----|------|-------|
| 方案 1 | FP16 + A100 80GB | 70 GB | 28 tokens/s | 高 | ⭐⭐ |
| 方案 2 | INT8 + A100 80GB | 35 GB | 55 tokens/s | 高 | ⭐⭐⭐⭐ |
| 方案 3 | INT4 + RTX 4090 | 18 GB | 36 tokens/s | **中** | ⭐⭐⭐⭐⭐ |
| 方案 4 | INT4 + H100 | 18 GB | 130 tokens/s | 极高 | ⭐⭐⭐ |

**结论**：**方案 3（INT4 + RTX 4090）性价比最高**
- ✅ 满足速度要求（36 > 30 tokens/s）
- ✅ 显存充足（4090 24GB > 18GB 模型 + 6GB KV/运行时）
- ✅ 成本可控

## 6. 优化推理速度的技巧

### 6.1 量化（Quantization）

**推荐**：
- **生产环境**：INT8 或 INT4（性价比最高）
- **研究/高精度**：FP16 或 BF16
- **移动端**：INT4 甚至更低

**量化方法对比**：

| 方法 | 速度 | 精度 | 难度 | 推荐 |
|-----|------|------|------|------|
| **GPTQ** | 快 | 高 | 中 | ⭐⭐⭐⭐⭐ |
| **AWQ** | 最快 | 中 | 低 | ⭐⭐⭐⭐ |
| **GGUF** | 中 | 高 | 低 | ⭐⭐⭐⭐（CPU推理） |

---

### 6.2 Flash Attention

**原理**：优化注意力计算，减少内存访问

**效果**：
- Prefill 速度提升 **2-4 倍**
- 支持更长的序列
- 显存占用减少 **20-30%**

**使用**：
```python
# transformers 库自动启用
model = AutoModelForCausalLM.from_pretrained(
    "Qwen/Qwen2.5-7B",
    torch_dtype=torch.float16,
    attn_implementation="flash_attention_2"  # 启用 Flash Attention
)
```

---

### 6.3 KV Cache 优化

**问题**：长对话时 KV Cache 占用大量显存

**解决方案**：

#### 1. PagedAttention（vLLM）
- 动态分页管理 KV Cache
- 显存利用率提升 **2-3 倍**

#### 2. Multi-Query Attention (MQA)
- 多个查询头共享 KV
- KV Cache 减少 **80%**

#### 3. Grouped-Query Attention (GQA)
- MQA 和传统注意力的折中
- KV Cache 减少 **50-70%**

---

### 6.4 批处理（Batching）

**连续批处理**（Continuous Batching）：
- 不同长度的请求可以在同一批次
- 吞吐量提升 **2-3 倍**

**工具**：
- **vLLM**：自动批处理 + PagedAttention
- **TensorRT-LLM**：NVIDIA 官方推理引擎
- **Text Generation Inference (TGI)**：HuggingFace 推理服务

---

### 6.5 模型并行（Model Parallelism）

**张量并行**（Tensor Parallelism）：
- 将单层权重切分到多张 GPU
- 适用：**单个模型太大**无法放入单卡

**流水线并行**（Pipeline Parallelism）：
- 将不同层放到不同 GPU
- 适用：**层数多**但单层不大

**示例**：
```
Qwen3.5-72B FP16 = 144 GB
→ 需要 2 × A100 80GB（张量并行）
→ 或 4 × A100 40GB（流水线并行）
```

## 8. 记忆卡：关键公式速查

### 🔥 最重要的公式

```
生成速度 (tokens/秒) = 内存带宽 (GB/s) / 模型大小 (GB) × 0.6
```

### 📊 模型大小

```
模型大小 (GB) = 参数量 (B) × 精度字节数

FP16: ×2    INT8: ×1    INT4: ×0.5
```

### ⚡ 两阶段时间

```
Prefill 时间 = (2 × 参数量 × 输入长度) / GPU算力
Decode 时间 = 输出长度 / 生成速度
```

### 🎯 显存需求

```
总显存 = 模型大小 + KV_Cache + 运行时显存
       = 模型大小 × (1 + 0.05×seq_len/1000 + 0.15)
```

## 总结

**推理速度的核心瓶颈是内存带宽，而非算力！**

记住三个关键点：
1. **Decode 阶段是瓶颈**：受内存带宽限制
2. **量化是提速捷径**：INT4 可提速 4-6 倍
3. **长序列会变慢**：KV Cache 随序列增长

**优化策略优先级**：
1. 量化（INT8/INT4）← 最高优先级
2. Flash Attention
3. vLLM/TGI 推理框架
4. KV Cache 优化
5. 多 GPU 并行

## 智能客服
智能客服用LLM代替传统的asr小模型时遇到的最大挑战是推理速度，要求首包在300ms中，团队做的工作：
1. 选择精度更低的量化模型 从官方的 Qwen/Qwen3.6-35B-A3B-FP8 切换为  palmfuture/Qwen3.6-35B-A3B-GPTQ-Int4；
2. 选择算力更高的模型 从H20切换为H200
3. 选择优化更好的推理框架 从sgLang切换为vLLM 
4. 使用Flash Attention (Flash Infer)
经过这一轮调整，整句输出首包从800ms优化到300ms
