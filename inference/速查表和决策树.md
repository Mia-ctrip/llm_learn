# 显存估算速查表和决策树

## 一句话快速判断

| 场景 | 判断方法 | 结论 |
|------|--------|------|
| **有显卡，选模型** | 显卡容量 × 80% = 最大可用显存 | 用"总显存速查表"反向查，选最大的模型 |
| **有模型，选显卡** | 参数量 × 2.3 + 0.8 = 所需GB | 查"GPU规格表"选第一个 ≥ 该值的卡 |
| **想支持大batch** | 每增加1个batch，KV Cache +0.8GB | 往上升一档显卡 |
| **KV Cache不够？** | 启用"Flash Attention"或量化 | 显存可减少50-70% |

## 总显存速查表（FP16，Batch=1，Seq=2K）

一眼看出"模型+显卡"的配置合不合理：

| 模型 | 显存需求 | T4(16GB) | V100(32GB) | A100(40GB) | L20(48GB) | H20(96GB) |
|------|---------|----------|-----------|-----------|----------|----------|
| 7B | ~17GB | ⚠️ 刚好 | ✓ 充足 | ✓ 充足 | ✓ 充足 | ✓ 充足 |
| 13B | ~31GB | ✗ 超过 | ⚠️ 刚好 | ✓ 充足 | ✓ 充足 | ✓ 充足 |
| 14B | ~33GB | ✗ 超过 | ✗ 超过 | ⚠️ 刚好 | ✓ 充足 | ✓ 充足 |
| 27B | ~63GB | ✗ | ✗ | ✗ | ✗ | ⚠️ 刚好 |
| 72B | ~167GB | ✗ | ✗ | ✗ | ✗ | ✗ 需要双卡 |

**符号说明**：
- ✓ 充足：能跑，而且有余量（推荐）
- ⚠️ 刚好：能跑，但没余地（不推荐生产环境）
- ✗ 超过：跑不了或需要优化

## 快速计算工作表

### 工作表 1：给定显卡，快速算出最大能跑的模型

**步骤**：

```
1. 写下你的显卡容量：________GB

2. 计算可用显存：________GB × 0.8 = ________GB（预留20%系统开销）

3. 根据下面的"参数量→显存需求"反查：

   参数量 B        显存需求（FP16, Batch=1）
   ─────────────────────────────────
   7B      →       ~17GB
   13B     →       ~31GB
   14B     →       ~33GB
   27B     →       ~63GB
   70B     →       ~160GB
   72B     →       ~167GB

4. 找到第一个 > 你的可用显存的，往上一个就是答案
```

**例子**：

- 你有 H20(96GB)
  - 可用：96 × 0.8 = 76.8GB
  - 查表：63GB(27B) ✓ 14B(33GB) ✓ 72B(167GB) ✗
  - **答案**：可以跑 27B，不能跑 72B

---

### 工作表 2：给定模型，快速算出需要什么显卡

**步骤**：

```
1. 写下模型参数量：________B

2. 写下精度（默认FP16）：________

3. 计算所需显存：
   
   a) 权重 = 参数量 × 精度字节数
      = ________B × ________ bytes = ________GB
   
   b) KV Cache (估算) = ________B × 2.3 / 10 ≈ ________GB
      （简化公式，粗略值）
   
   c) 运行时 = 权重 × 0.15 = ________GB
   
   d) 总计 = ______ + ______ + ______ = ________GB

4. 从下列GPU中选择第一个 ≥ 总计 的：
   
   T4(16GB) → V100(32GB) → A100(40GB) → L20(48GB) → H20(96GB)
   
5. 写下答案：需要 ________
```

**例子**：

- 模型：Qwen2.5-14B，精度：FP16
  ```
  a) 权重 = 14B × 2 bytes = 28GB
  b) KV Cache = 14 × 2.3 / 10 ≈ 3.2GB
  c) 运行时 = 28 × 0.15 = 4.2GB
  d) 总计 = 28 + 3.2 + 4.2 = 35.4GB
  e) 选GPU：T4(不够) → V100(不够) → A100(不够) → L20(不够) → H20(充足) ✓
  
  答案：需要 H20(96GB) 或 A100双卡
  ```

                        ↓                           ↓

## 精度选择指南

| 精度 | 显存 | 推理精度 | 推理速度 | 何时用 |
|------|------|--------|--------|--------|
| FP32 | 最多 | 最高 | 最慢 | 不推荐（推理） |
| FP16 | 中等 | 高 | 中等 | **推荐**（标准方案） |
| BF16 | 中等 | 高 | 中等 | 推荐（需要BF16硬件支持） |
| INT8 | -50% | 中 | 快 | 显存紧张时用 |
| INT4 | -75% | 低 | 最快 | 极度缺显存或边界计算 |

**精度选择决策树**：

```
显存充足？
  ├─ YES → FP16（标准推荐）
  ├─ 勉强 → INT8 量化（-50%显存）
  └─ 严重不足 → INT4 量化 (-75%显存) 或 单纯降模型规模
```

## 常见问题速答

### Q1：7B 模型为什么有时候用 T4(16GB)，有时候需要 L20(24GB)？

**A：可能的原因**：

- ✓ T4 情况：使用了 INT8 量化（7B × 1 = 7GB）+ Flash Attention
- ✗ L20 情况：使用了 FP16 精度，完整部署

---

### Q2：为什么 Qwen3 70B 有时候配 H20_1_96，有时候配 H20_8_768？

**A：显存需求不同**：

- H20_1_96：**单卡部署**，必须量化到 INT8
  - 70B × 1 bytes + cache + runtime ≈ 95GB ✓
  
- H20_8_768：**多卡并行部署**，完整 FP16
  - 用于支持更大 batch 或更长序列
  - 或用于高吞吐量服务（多个请求）

---

### Q3：我的实时推理服务，KV Cache 和 Batch Size 如何选？

**A：按场景选择**：

- **低延迟场景**（单用户）：Batch=1，Seq_len=越短越好
  - 例：在线聊天 → Batch=1, Seq_len=4K
  
- **高吞吐量**（多用户共享）：Batch=更大，Seq_len=预设
  - 例：批量评分 → Batch=32, Seq_len=2K
  
- **流式输出**：边生成边输出，动态扩展 KV Cache
  - 显存取决于最长可能的生成长度

---

### Q4：多卡部署如何分配显存？

**A：有两种方案**：

**方案 1：数据并行**（相同）
```
每卡存一份完整模型 + 不同 batch 的数据

总显存 = (模型权重 + 运行时) × 卡数 + KV_cache × batch

优点：容易实现
缺点：显存浪费
```

**方案 2：模型并行**（不同）
```
模型分割存在多卡上，每卡只存部分

总显存 = 模型权重 + KV_cache × batch + 通信开销

优点：支持超大模型
缺点：通信开销大，需要优化通信拓扑
```

**建议**：
- 模型 < 30B：用数据并行
- 模型 > 70B：用模型并行或张量并行
