# 综合学习总结 - 从0到精通

## 你学到了什么？

### 一、核心概念（必须理解）

#### 1. 参数量（用"B"表示）
- **7B** = 70亿个参数
- **14B** = 140亿个参数
- 数字越大，模型越大，计算量越多，需要的显存越多

#### 2. 精度（关键的显存乘数）
- **FP32**：4字节/参数（训练用，显存最多）
- **FP16**：2字节/参数（推理标准，显存中等）→ **你应该默认选这个**
- **INT8**：1字节/参数（量化推理，显存减半）
- **INT4**：0.5字节/参数（极限压缩，显存减75%）

#### 3. 显存的三部分
| 部分 | 含义 | 大小 | 是否固定 |
|------|------|------|--------|
| **权重** | 模型参数存储 | 参数量 × 精度 | 固定 |
| **KV Cache** | 推理时保存历史 | 随 batch 和序列长度增长 | 动态 |
| **运行时** | 临时计算显存 | 约权重的 15% | 半固定 |

## 二、三条必背的公式

### 公式 1：权重显存

```
权重显存 (GB) = 参数量(B) × 精度字节数

例子：
Qwen2.5-7B FP16 = 7B × 2 = 14GB
Qwen2.5-14B FP16 = 14B × 2 = 28GB
Qwen2.5-7B INT8 = 7B × 1 = 7GB  ← 量化的威力
```

### 公式 2：推理总显存（快速版）

```
总显存 (GB) ≈ 参数量(B) × 2.3 + KV_cache

或者更简化：
总显存 (GB) ≈ 参数量(B) × 2.5   （包括KV和运行时）

例子：
7B 模型：7 × 2.5 = 17.5GB ≈ T4(16GB)√
14B 模型：14 × 2.5 = 35GB ≈ L20(48GB)√
70B 模型：70 × 2.5 = 175GB ≈ H20双卡(192GB)√
```

### 公式 3：KV Cache 详细版（需要时用）

```
KV Cache (GB) = 2 × batch_size × seq_len × hidden_size × num_layers × precision_bytes / 1B

快速近似：
每个 batch、每个 token 的 KV Cache ≈ 1MB（对大多数 7B-14B 模型）

所以：
KV Cache ≈ batch_size × seq_len (tokens) / 1000 (GB)

例子：
batch=1, seq_len=2000 → KV ≈ 2/1000 = 0.002GB ≈ 2MB... 等等，这不对
实际更接近：
batch=1, 2000 tokens → KV ≈ 0.8GB（对 7B 模型）
```

## 三、快速决策树

### 我有一张 GPU，能跑什么模型？

```
显卡容量 × 0.8（预留20%） = 可用显存

查表找第一个 ≤ 可用显存的模型：
  7B:     17GB  ✓
  13B:    31GB  ?
  14B:    33GB  ?
  27B:    63GB  ✗
  70B:   160GB  ✗✗

例子：
  T4(16GB) × 0.8 = 12.8GB → 只能 7B（需要量化）
  L20(48GB) × 0.8 = 38.4GB → 可以 7B/13B/14B
  H20(96GB) × 0.8 = 76.8GB → 可以到 27B（需要量化的 70B）
```

### 我想跑一个模型，需要什么 GPU？

```
估算所需显存：参数量 × 2.5（GB）

找第一个 ≥ 该值的 GPU：
  ≥ 7GB:     T4(16GB)
  ≥ 17GB:    V100(32GB)
  ≥ 33GB:    A100(40GB) / L20(48GB)
  ≥ 90GB:    H20(96GB)
  ≥ 150GB:   2×H20 或 H20(96GB) + INT8 量化
```

## 四、实战技巧

### 技巧 1：遇到显存不足时的优先级

```
1. 降精度（FP16 → INT8）      成本：-50% 显存，精度略降
2. 启用 Flash Attention       成本：-30% 显存，速度快
3. 量化（INT8 → INT4）       成本：-75% 显存，精度明显降
4. 多卡分布式               成本：需要更多硬件
5. 降模型规模                成本：能力下降
```

### 技巧 2：充分利用显存余量

```
如果显存利用率 < 50%（很浪费）：
  ✓ 增加 batch_size（支持更多并发）
  ✓ 增加 seq_len（支持更长输入）
  ✓ 混合部署其他小模型（embedding）

如果显存利用率 60-80%（最理想）：
  ✓ 这就是你的数据中 L20_1_48 的配置
  
如果显存利用率 > 90%（太紧张）：
  ✗ 风险高，可能 OOM
  ⚠️  需要升级或优化
```

### 技巧 3：量化的实战使用

```
学会用 GPTQ / AWQ 量化：
  原始：Qwen2.5-14B（28GB）
  INT8：Qwen2.5-14B（14GB）
  GPTQ：Qwen2.5-14B（8GB）← 你的数据中使用过

框架推荐：
  ✓ ollama（最简单，一键量化）
  ✓ vLLM（高吞吐量推理）
  ✓ transformers（灵活，学习用）
```

## 六、最常见的错误

| 错误 | 为什么错 | 正确做法 |
|------|--------|--------|
| "模型参数量大 = 一定需要大显存" | 忽视了精度的作用 | 先看精度，FP16 用 INT8 救 50% 显存 |
| "买最贵的 GPU 最安全" | 浪费钱 | 算好显存需求，选最小满足的 GPU |
| "KV Cache 可以忽略不计" | 长序列时会溅出 | Batch+序列增长时必须考虑 |
| "所有模型都用 FP32" | 显存爆炸 | 推理用 FP16，训练才考虑混合精度 |
| "单卡承载不了就放弃" | 可以量化或多卡 | 试试 INT8 或使用 Tensor Parallelism |

## 七、对应用场景的建议

### 场景 A：在线聊天机器人（低延迟）

```
需求：快速响应，用户体验好
推荐配置：
  模型：Qwen2.5-7B 或 Qwen3-7B
  精度：FP16
  硬件：L20_1_48（足够，价格合理）
  优化：启用 Flash Attention
  
显存预期：~17GB（占 35% 容量，很富裕）
```

### 场景 B：智能客服（高吞吐量）

```
需求：同时处理多个用户请求
推荐配置：
  模型：Qwen2.5-14B（更准确）
  精度：FP16
  硬件：L20_1_48 或 H20
  优化：增加 Batch_size=4-8
  
显存计算：
  权重：28GB
  KV Cache（batch=8, 2k）：0.8 × 8 = 6.4GB
  运行时：4GB
  总计：38.4GB < 48GB ✓
```

### 场景 C：离线批处理（成本优先）

```
需求：处理大量数据，不在乎速度
推荐配置：
  模型：Qwen2.5-7B INT8（降成本）
  精度：INT8
  硬件：T4 × 多张（便宜）
  优化：最大化 Batch_size

显存计算：
  权重：7GB
  KV Cache（batch=32, 2k）：0.8 × 32 = 25.6GB
  运行时：1.5GB
  总计：34GB < 48GB ✓ 或用 2×T4
```

### 场景 D：生成式任务（图生成）

```
需求：Stable Diffusion、视频生成
推荐配置：
  模型：Stable Diffusion 3（>3GB）+ 其他模型
  精度：FP16
  硬件：L20_1_48（足够）
  优化：可并行多个生成任务
  
显存预期：
  模型权重：~3-5GB
  中间结果：~10GB（高分辨率时增加）
  总计：~15GB < 48GB ✓
```
