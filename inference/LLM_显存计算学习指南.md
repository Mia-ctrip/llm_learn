# LLM 显存计算学习指南

## 1. 模型大小定义

### 1.1 参数量（Parameters）

**B = Billion = 十亿**

- **7B** = 70亿参数
- **13B** = 130亿参数  
- **14B** = 140亿参数
- **27B** = 270亿参数
- **35B** = 350亿参数
- **72B** = 720亿参数

### 1.2 如何理解参数量

模型的参数就像神经网络中的"权重"，每个参数都需要存储和计算。

参数量 = 所有权重矩阵中元素的总数

常见的 Transformer 模型参数量估算：
- **Embedding层**：vocab_size × hidden_dim
- **Attention层**：每层大约 hidden_dim²
- **MLP层**：每层大约 hidden_dim × 4 × hidden_dim
- 总参数 ≈ 12 × hidden_dim × num_layers

**例子**：
- Qwen2.5-7B 有 **70亿个参数**
- Qwen2.5-14B 有 **140亿个参数**（大约是7B的2倍）

---

## 2. 模型精度（数据类型）
模型的精度就是模型权重的精度

### 2.1 常见精度类型

| 精度 | 字节数 | 能表示的范围 | 备注 |
|------|--------|-----------|------|
| **FP32** (float32) | 4 字节 | ±3.4×10³⁸ | 全精度，精度最高，显存最多 |
| **FP16** (float16) | 2 字节 | ±65504 | 半精度，可能精度丧失 |
| **BF16** (bfloat16) | 2 字节 | ±3.4×10³⁸ | 谷歌提出，兼具FP32范围和FP16效率 |
| **INT8** (int8) | 1 字节 | -128 ~ 127 | 量化，显存最少，推理速度快 |
| **FP8** (float8) | 1 字节 | ±240 | 超轻量化 |

### 2.2 精度对显存的影响

**模型权重占用的显存 = 参数量 × 精度字节数**

例子：以Qwen2.5-7B为例（70亿参数）

- **FP32 精度**：7B × 4 字节 = **28GB**
- **FP16 精度**：7B × 2 字节 = **14GB**
- **INT8 精度**：7B × 1 字节 = **7GB**
- **FP8 精度**：7B × 1 字节 = **7GB**

### 2.3 精度选择建议

- **训练**：通常用 FP32 或混合精度 (FP16 + FP32)
- **推理**：可以用 FP16、BF16、INT8、FP8
- **移动端**：INT8、FP8

---

## 3. 推理时的显存组成

推理时总显存 = **模型权重 + KV Cache + 运行时显存**

### 3.1 模型权重（Model Weights）

**公式**：权重显存 = 参数量 × 精度字节数

这是**固定的**，和输入长度、batch size 无关。

**例子**：
- Qwen2.5-7B (FP16) = 7B × 2 = 14GB
- Qwen2.5-14B (FP16) = 14B × 2 = 28GB

### 3.2 KV Cache（关键值缓存）

#### 什么是 KV Cache？

在 Transformer 中，Attention 机制需要计算当前 token 与所有历史 token 的相似度。
- **K** = Key （历史 token 的查询键）
- **V** = Value （历史 token 的特征向量）

每生成一个 token，都要保存该 token 的 K 和 V，这就是 KV Cache。

#### KV Cache 显存计算

**公式**：
```
KV Cache 显存 = 2 × (batch_size) × (sequence_length) × (hidden_dim) × (num_layers) × (精度字节数)
              = 2 × batch × seq_len × hidden_dim × num_layers × bytes_per_token

简化公式：
KV Cache 显存 ≈ 2 × 参数量 × (seq_len / hidden_size)

实用公式：
KV Cache 显存 ≈ batch_size × seq_len × 参数量 × 精度字节数 / (1000M × hidden_size系数)
```

#### 实际计算示例

**Qwen2.5-7B**
- 参数量：7B
- hidden_size：3584
- num_layers：28
- 精度：FP16 (2字节)

**单个 token 的 KV Cache**：
```
= 2 × 1(batch) × 1(token) × 3584(hidden) × 28(layers) × 2(bytes)
= 2 × 3584 × 28 × 2
≈ 0.4 MB
```

**1000个 tokens 的 KV Cache**：
```
= 2 × 1 × 1000 × 3584 × 28 × 2
≈ 403 MB
```

**2K tokens (2000) 的 KV Cache**：
```
= 2 × 1 × 2000 × 3584 × 28 × 2
≈ 806 MB ≈ 0.8GB
```

**4K tokens (4000) 的 KV Cache**：
```
= 2 × 1 × 4000 × 3584 × 28 × 2
≈ 1.6 GB
```

**Batch=4, 2K tokens 的 KV Cache**：
```
= 2 × 4 × 2000 × 3584 × 28 × 2
≈ 3.2 GB
```

#### 快速记忆法

对于一般的大模型（参数量为 P，以 B 计），**单个 batch 的 KV Cache**：

```
KV Cache (GB) ≈ seq_len × batch_size × 2 / 1000

例如：
- 1个batch，2K tokens，7B模型 → KV Cache ≈ 0.8 GB
- 4个batch，2K tokens，7B模型 → KV Cache ≈ 3.2 GB
```

### 3.3 运行时显存（Runtime Memory / 激活值）

临时显存，用于存储计算中间结果。推理时不需要梯度，激活值比训练时少很多。

#### 快速估算

```
运行时显存 ≈ 模型权重的 10-20%
```

#### 详细分析：Prefill vs Decode 的激活值

运行时激活值在两个阶段差异很大，**取峰值**：

```
Prefill 阶段（激活值最大）：
  主要消耗：
  ① Attention 分数矩阵: batch × heads × seq_len² × bytes
    例 (seq_len=2048, 32头, FP16): 32 × 2048² × 2 ≈ 256 MB
  ② LM Head logits:      batch × seq_len × vocab_size × bytes
    例 (seq_len=2048, vocab=152064, FP16): 2048 × 152064 × 2 ≈ 593 MB ← 往往是最大的！
  ③ FFN 中间激活:        batch × seq_len × 4×hidden_dim × bytes
    例: 2048 × 3584 × 4 × 2 ≈ 56 MB

Decode 阶段（激活值很小）：
  每次只处理 1 个 token，所有激活都很小：
  ① Q 向量:      (batch, 1, hidden_dim) ≈ 7 KB
  ② logits:      (batch, 1, vocab_size) ≈ 300 KB
  ③ FFN 中间值:  (batch, 1, 4×hidden_dim) ≈ 28 KB
  → 相比 KV Cache 可以忽略
```

#### 激活值峰值的简化计算

```
Prefill 激活峰值 ≈ max(attention分数矩阵, logits, FFN激活)
                ≈ batch × seq_len × max(heads × seq_len, vocab_size) × bytes

当 vocab_size > heads × seq_len 时，logits 是激活值的大头
当 heads × seq_len > vocab_size 时，attention 分数矩阵是大头

实际部署中：Prefill 激活是临时的（算完即释放），不持续占用显存
           真正持续累积的是 KV Cache，才是长期显存压力
```

#### 常见误区

```
❌ 误解：运行时激活值主要来自 input_embedding 或单个 token 的 logits
✅ 正确：Prefill 时最大的是 LM Head 输出 (seq_len × vocab_size)
         和 attention 分数矩阵 (heads × seq_len²)
         Decode 时激活值很小，KV Cache 才是显存主力
```

## 4. 推理显存总计公式

```
总显存 (GB) = 权重显存 + KV_Cache + 运行时显存(峰值)

详细版本：
总显存 = (参数量 × 精度字节数) 
       + (2 × batch × seq_len × hidden_dim × layers × 精度字节数 / 1B)
       + max(Prefill激活峰值, Decode激活峰值)

生产环境简化估算：
总显存 ≈ 权重显存 + KV_Cache(max_output_len) + ~10% 余量
因为：Prefill 激活是临时的，Decode 激活很小，KV Cache 才是长期压力
```

### 4.1 实例计算：Qwen2.5-7B FP16

**配置**：
- 模型：Qwen2.5-7B
- 精度：FP16（2字节）
- Batch size：1
- Sequence length：2000 tokens
- hidden_dim：3584
- num_layers：28

**计算**：

1. **权重**：7B × 2 = **14 GB**

2. **KV Cache**：
   ```
   = 2 × 1 × 2000 × 3584 × 28 × 2 / (1000M)
   ≈ 0.8 GB
   ```

3. **运行时**：14 × 0.15 = **2.1 GB**

4. **总计**：14 + 0.8 + 2.1 = **16.9 GB** → **≈ 17 GB**

### 4.2 实例计算：Qwen2.5-14B FP16

**配置**：同上，但模型是14B

1. **权重**：14B × 2 = **28 GB**

2. **KV Cache**：
   ```
   = 2 × 1 × 2000 × 3584 × 28 × 2 / (1000M)
   ≈ 0.8 GB  （这个不变，因为hidden_dim相同）
   ```

3. **运行时**：28 × 0.15 = **4.2 GB**

4. **总计**：28 + 0.8 + 4.2 = **33 GB** → 需要 **2 × 16GB显卡** 或 **1 × 40GB显卡**

# 6. 训练时的显存需求

### 6.1 训练显存的组成

训练比推理多出的核心开销：**梯度 + 优化器状态 + 前向激活值（用于反向传播）**

```
训练总显存 = 模型权重 + 梯度 + FP32权重副本 + 优化器状态 + 激活值
```

### 6.2 逐项拆解（BF16 混合精度 + AdamW 优化器）

| 组成部分 | 公式 | 每参数字节 | 说明 |
|---------|------|-----------|------|
| **模型权重** | P × 2 | 2 bytes | BF16 存储 |
| **梯度** | P × 2 | 2 bytes | BF16 梯度 |
| **FP32 权重副本** | P × 4 | 4 bytes | AdamW 需要 FP32 主权重做更新 |
| **优化器状态（AdamW）** | P × 8 | 8 bytes | 一阶动量(4B) + 二阶动量(4B) |
| **激活值** | 取决于 batch×seq_len×hidden | — | 反向传播需要，不能释放 |

**核心公式：**

```
训练显存（不含激活值）= P × (2 + 2 + 4 + 8) = P × 16 bytes
训练显存（含激活值）  ≈ P × 16~18 bytes

速算口诀：训练显存 ≈ 参数量(B) × 16~18 GB
```

### 6.3 实例计算：27B 模型 BF16 训练

假设：BF16 混合精度，AdamW 优化器

| 部分 | 计算 | 大小 |
|------|------|------|
| 模型权重 (BF16) | 27B × 2 | **54 GB** |
| 梯度 (BF16) | 27B × 2 | **54 GB** |
| FP32 权重副本 | 27B × 4 | **108 GB** |
| 优化器状态 | 27B × 8 | **216 GB** |
| **小计（不含激活值）** | | **432 GB** |
| 激活值（batch=1, seq=2K 估算） | | **~20 GB** |
| **总计** | | **≈ 450 GB** |

### 6.4 推理 vs 训练显存对比（27B, BF16）

| 场景 | 显存 | 倍数关系 |
|------|------|----------|
| **推理** (BF16) | ~63 GB | 1x |
| **训练** (BF16+AdamW) | ~450 GB | **~7x** |

→ 训练 27B 模型通常需要 **4~8 张 A100-80GB** 或 **2~4 张 H100-80GB**

### 6.5 不同优化器的训练显存对比

| 优化器 | 优化器状态 | 每参数总字节 | 27B 模型总显存 |
|--------|-----------|------------|---------------|
| **SGD（无动量）** | 无 | 2+2 = 4 | ~108 GB |
| **SGD+动量** | P × 4 | 2+2+4 = 8 | ~216 GB |
| **Adam / AdamW** | P × 8 | 2+2+4+8 = 16 | ~432 GB |
| **8-bit AdamW** | P × 2 | 2+2+4+2 = 10 | ~270 GB |

### 6.6 降低训练显存的常用手段

| 方法 | 效果 | 说明 |
|------|------|------|
| **LoRA / QLoRA** | 只训练 1~5% 参数 | 27B 模型只需 ~60~80 GB 即可微调 |
| **DeepSpeed ZeRO-3** | 参数/梯度/优化器切分到多卡 | 多卡均摊，单卡显存大幅降低 |
| **8-bit AdamW** | 优化器状态 8B → 2B | 节省约 162 GB |
| **Gradient Checkpointing** | 激活值大幅减少 | 用重算换显存，速度慢 ~20% |
| **全量化微调 (QLoRA)** | 权重 INT4 + LoRA | 27B 约 24GB 即可微调 |

## 8. 实战步骤：给定显卡，快速确定能跑的模型

### 情景：你有一张 24GB L20 显卡，想选模型

**步骤**：

1. **预留系统开销**：24GB × 80% = **19.2GB 可用**

2. **假设常见配置**：
   - 精度：FP16
   - Batch size：1
   - Seq_len：2000 tokens

3. **逐个检验**：
   - 7B：14 + 0.8 + 2 = 16.8GB ✓ **可以**
   - 13B：26 + 0.8 + 4 = 30.8GB ✗ **超了**

4. **结论**：L20(24GB) 适合 **7B 模型**

### 情景：你想跑 Qwen2.5-14B，需要什么显卡？

**步骤**：

1. **计算显存需求**（FP16, Batch=1）：
   - 权重：28GB
   - KV Cache：0.8GB
   - 运行时：4.2GB
   - **总计：33GB**

2. **选择显卡**：
   - 单卡方案：需要 **≥40GB** → A100 或 H20
   - 双卡方案：2×T4(32GB) **刚好** 或 2×L20(48GB) **充足**

3. **推荐**：选择 A100_1_40 或 H20_1_96（更富裕）

```
INT8 推理 ≈ P(B) × 1.2

```

## 11. 并发请求与显存的关系

### 11.1 核心认知：QPM/QPS 如何影响显存

```
推理显存 = 模型权重（固定，和请求量无关） + KV Cache（随并发增长） + 运行时激活（随并发增长）
```

- **权重**：加载一次，和请求量 **无关**
- **KV Cache**：每个 **同时在处理的请求** 都独占一份，和 **并发数直接相关**
- **运行时激活**：同理，并发越高越大

QPM/QPS 不直接占显存，但它决定了 **同一时刻有多少请求在同时推理**：

```
并发数 ≈ QPS × 单请求平均处理时间(秒)
```

### 11.2 为什么单模型能处理高并发？—— Continuous Batching

> **串行的是"token"，不是"请求"**

每个 Decode step，推理引擎把 **所有正在进行的请求** 打包成一个 batch 一起算：

```
时间线：
Step 1: [请求A的第1个token, 请求B的第1个token, 请求C的第1个token] → 一次前向
Step 2: [请求A的第2个token, 请求B的第2个token, 请求C的第2个token] → 一次前向
Step 3: [请求A的第3个token, 请求B的EOS结束!, 请求C的第3个token] → B结束
Step 4: [请求A的第4个token, 新请求D插入!,    请求C的第4个token] → D补位
```

**并发能力的关键：**
- 单个请求内部：串行（第 n+1 个 token 依赖第 n 个）
- 多个请求之间：并行（同一个 step 里 batch 一起算，GPU 擅长这个）
- 某请求生成完（EOS），新请求 **动态插入**，不浪费 GPU

主流推理引擎（vLLM、TensorRT-LLM、TGI）都支持 Continuous Batching。

### 11.3 实战：从 QPM 估算显存需求

**案例：3B Qwen 模型，QPM峰值 300，日请求 20000**

**Step 1：估算并发数**
```
QPS = 300 / 60 = 5 req/s
假设单请求平均处理 2 秒
并发数 ≈ 5 × 2 = 10 个请求同时在跑
```

**Step 2：计算单实例显存**

| 部分 | 计算 | 大小 |
|------|------|------|
| 权重 (BF16) | 3B × 2 | 6 GB |
| KV Cache (batch=10, seq=2K) | 2×10×2048×hidden×layers×2 | ~0.5 GB |
| 运行时激活 | | ~1 GB |
| **单实例总计** | | **≈ 7.5 GB** |

**Step 3：生产环境选型**

| 因素 | 说明 |
|------|------|
| **冗余** | 不能跑满，留 30% 余量 |
| **高可用** | 至少 2 个副本，一个挂了还有备用 |
| **吞吐** | 单卡 batch=10 不一定能满足 QPS=5 的吞吐 |

**推荐方案：**

| 方案 | 配置 | 说明 |
|------|------|------|
| 最低配 | 2 × T4 (16GB) | 每卡一个实例，batch=10 够用 |
| 推荐 | 2 × L20 (24GB) | 余量充足，可应对突发 |
| 稳妥 | 3 × T4 (16GB) | 2个服务 + 1个冗余 |

> 日请求 20000 其实不算大（平均 ~0.23 QPS），**峰值 QPM 才是选型的决定因素**。
