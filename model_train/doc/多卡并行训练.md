# 多卡并行训练：策略、实现与实验观察

> 学习日期：2026-07-07  
> 实验环境：2× NVIDIA H20 96GB（NVLink 900GB/s）  
> 实验模型：mini_gpt（embed=256, heads=4, layers=10, ~25M 参数）  
> 框架：PyTorch 2.9.0

---

## 一、为什么需要多卡

单卡训练的瓶颈通常有两个：
1. **显存不够**：模型参数 + 梯度 + 优化器状态 + 激活值超出单卡容量
2. **速度太慢**：单卡算力不足以在合理时间内完成训练

多卡通过拆分工作来突破这两个瓶颈。但多卡不是"无脑加速"——引入跨卡通信后，收益取决于模型规模和并行策略的选择。

---

## 二、并行策略全景图

```
多卡并行策略
├── 数据并行（Data Parallelism）—— 模型不变，数据切分
│   ├── DataParallel (DP)          —— PyTorch 内置，多线程，简单易用
│   └── DistributedDataParallel (DDP) —— 多进程，工业标准
│
├── 模型并行（Model Parallelism）—— 数据不变，模型切分
│   ├── Tensor Parallelism (TP)    —— 单层内部矩阵切分到多卡
│   └── Pipeline Parallelism (PP)  —— 不同层分配到不同卡
│
└── 混合并行 —— 以上策略的组合（大模型必选）
    └── 3D Parallelism = DP + TP + PP（如 Megatron-LM、DeepSpeed）
```

### 选择原则

| 模型规模 | 推荐策略 | 说明 |
|---------|---------|------|
| < 1B 参数，单卡放得下 | **DDP**（数据并行） | 每卡一份完整模型，数据切分 |
| 1B~10B，单卡放不下 | **DDP + TP** | 模型层内切分 + 数据并行 |
| > 10B | **3D 并行 (DP + TP + PP)** | 必须切分模型，需要框架支持 |
| 推理超大模型 | **TP + PP** | 推理不需要梯度，Pipeline 气泡可接受 |

## 三、DataParallel (DP)

### 3.1 工作原理

```
              GPU 0 (主卡)                    GPU 1 (从卡)
              ────────────                    ────────────
scatter       接收全部 batch，切分发送         只接收自己那份
replicate     参数在自己身上                   接收参数副本
forward       正常计算                         正常计算
gather        收集所有卡的输出 ←────────────── 发送输出给 GPU0
loss 计算     ✅ 在 GPU0 上算                  ❌ 不参与
backward      正常计算梯度                     正常计算梯度
gather grad   收集所有梯度 ←───────────────── 发送梯度给 GPU0
optimizer     ✅ 在 GPU0 上更新参数             ❌ 不参与
broadcast     广播新参数 ────────────────────→ 接收新参数
```

### 3.2 代码（只需 1 行）

```python
model.to(device)
if torch.cuda.device_count() > 1:
    model = nn.DataParallel(model)  # 自动拆分 batch 到多卡
```

### 3.3 DP 的固有缺陷

| 缺陷 | 原因 | 影响 |
|------|------|------|
| **GIL 限制** | Python 多线程，GIL 导致线程不能真正并行 | 2 卡收益还行，4 卡开始打折扣 |
| **GPU0 不对称** | loss 计算、梯度汇总、参数更新都在 GPU0 | GPU0 更忙，利用率比 GPU1 高 |
| **每 step 重复 replicate** | 每次前向都要复制模型参数到各卡 | 额外通信开销 |
| **gather 瓶颈** | 所有输出汇聚到 GPU0，GPU0 显存压力大 | 大 batch 时 GPU0 容易 OOM |

### 3.4 GIL 限制是什么

**GIL = Global Interpreter Lock（全局解释器锁）**，是 Python 的一个机制：同一时刻只允许一个线程执行 Python 代码。

DP 使用多线程实现多卡并行，每个 GPU 一个线程。但这些线程都要执行 Python 代码（forward/backward/loss 等），而 GIL 只有一把：

```
线程 0 (GPU 0): ████░░░░████░░░░████  断断续续
线程 1 (GPU 1): ░░░░████░░░░████░░░░  交替执行
线程 2 (GPU 2): ░░░░░░░░░░░░████░░░░  排队等待
                  ↑
              同一时刻只有 1 个线程在跑 → 假并行
```

所以 DP 的 4 卡加速比不是 4x 而是约 2.5x——线程越多，GIL 竞争越严重。

DDP 用多进程（每个进程有自己的 GIL），所以不受这个限制，4 卡接近 3.8x 加速比。

### 3.5 什么时候用 DP

- 快速实验、不想改代码
- 2 卡以内、模型较小
- Windows 环境（DDP 在 Windows 上有兼容问题）

**其他所有场景都应该用 DDP。**

## 四、DistributedDataParallel (DDP)

### 4.1 工作原理

```
每个 GPU 对应一个独立的 Python 进程
┌──────────────┐     ┌──────────────┐
│  进程 0      │     │  进程 1      │
│  GPU 0       │     │  GPU 1       │
│              │     │              │
│  模型副本 0  │     │  模型副本 1  │
│  数据子集 0  │     │  数据子集 1  │
│              │     │              │
│  forward     │     │  forward     │
│  backward    │◄───►│  backward    │  ← all-reduce 梯度同步
│  optimizer   │     │  optimizer   │  ← 用相同梯度更新，保持一致
└──────────────┘     └──────────────┘
```

与 DP 的核心区别：
- **多进程**而非多线程，没有 GIL 限制
- **all-reduce** 梯度同步，所有卡参与梯度汇总（不是只汇聚到 GPU0）
- **不复制模型**，每张卡始终持有自己的完整模型副本
- 各卡**独立计算**，只在反向传播时同步梯度

#### 单卡显存占用构成

DDP 下每张卡需要的显存：

```
单卡显存占用
├── 模型参数（FP32）          ← 完整副本
├── 梯度（FP32）              ← 完整副本
├── 优化器状态（Adam）        ← 动量 + 方差，2 份完整副本
└── 激活值（中间计算）        ← 只存当前 batch 的，和 batch_size 相关
```

**以 25M 参数模型（FP32）为例**：
| 项目 | 占用 |
|------|------|
| 模型参数 | ~100 MB |
| 梯度 | ~100 MB |
| 优化器状态（Adam） | ~200 MB |
| 激活值 | 随 batch_size 变化 |
| **总计（不含激活值）** | **~400 MB** |

**DDP 的前提**：单卡能放下完整的模型 + 梯度 + 优化器状态。Adam 优化器状态容易忽略——它要存 2 份额外的参数大小。

### 4.2 代码改动要点

```python
import torch.distributed as dist
from torch.nn.parallel import DistributedDataParallel as DDP
from torch.utils.data.distributed import DistributedSampler

# 1. 初始化进程组（torchrun 自动设置环境变量）
dist.init_process_group(backend='nccl')
local_rank = int(os.environ['LOCAL_RANK'])
torch.cuda.set_device(local_rank)

# 2. 用 DistributedSampler 切分数据
sampler = DistributedSampler(dataset, num_replicas=world_size, rank=rank, shuffle=True)
dataLoader = DataLoader(dataset, batch_size=batch_size_per_gpu, sampler=sampler, ...)

# 3. DDP 包装模型
model = model.to(device)
model = DDP(model, device_ids=[local_rank])

# 4. 每个 epoch 重新打乱
sampler.set_epoch(epoch)

# 5. 保存时用 model.module 去掉 module. 前缀
torch.save(model.module.state_dict(), 'model.pth')
```

### 4.3 启动方式

```bash
# 必须用 torchrun 启动，不能直接 python
torchrun --nproc_per_node=2 train_ddp.py    # 2 卡
torchrun --nproc_per_node=4 train_ddp.py    # 4 卡
```

`torchrun` 会自动设置 `LOCAL_RANK`、`RANK`、`WORLD_SIZE`、`MASTER_ADDR`、`MASTER_PORT` 等环境变量。

### 4.4 DDP 跨卡通信机制

DDP 的核心通信操作是 **all-reduce**，发生在 `loss.backward()` 阶段：

```
梯度计算完成
    ↓
all-reduce: 所有卡的梯度求和取平均
    ↓
每张卡用相同的平均梯度更新参数
    ↓
所有卡的模型保持一致
```

#### all-reduce 在哪里执行

all-reduce 的求和操作是在**每张卡各自的 GPU SM 上执行**的，通过运行 NCCL Kernel（一个 CUDA 程序）来完成：

```
GPU 0 的 SM                GPU 1 的 SM
┌─────────────┐           ┌─────────────┐
│ 执行 NCCL   │           │ 执行 NCCL   │
│ Kernel      │           │ Kernel      │
│             │           │             │
│ 接收数据    │           │ 接收数据    │
│ 求和操作    │  ← NVLink →  求和操作    │
│ 发送数据    │           │ 发送数据    │
└─────────────┘           └─────────────┘
```

- NCCL Kernel 是软件（CUDA 程序），不是 SM 的硬件部分
- 每张卡都运行自己的 NCCL Kernel，在自己的 SM 上执行求和
- 通过 NVLink/PCIe 传输数据，边传输边累加
- 没有中心化的主卡，是去中心化的协作计算

#### Ring AllReduce 算法

NCCL 使用 Ring AllReduce 实现高效的多卡梯度同步：

```
以 4 卡为例，梯度分成 4 块：

初始：
GPU 0: [a0, b0, c0, d0]    GPU 2: [a2, b2, c2, d2]
GPU 1: [a1, b1, c1, d1]    GPU 3: [a3, b3, c3, d3]

===== 阶段 1: Reduce-Scatter（边传边累加）=====

每张卡发一块给下一张，收到后累加：
Step 1: GPU 0 发 d0 → GPU 1，GPU 1 计算 d0 + d1
Step 2: 继续传递和累加...
Step 3: 继续传递和累加...

阶段 1 结束：每张卡有一块完整求和
GPU 0: [_, _, c_sum, _]    GPU 2: [a_sum, _, _, _]
GPU 1: [_, _, _, d_sum]    GPU 3: [_, b_sum, _, _]

===== 阶段 2: AllGather（分发完整结果）=====

每张卡把自己那块求和结果在 ring 中传递，其他卡接收并保存。

阶段 2 结束：每张卡都有全部求和结果
GPU 0: [a_sum, b_sum, c_sum, d_sum]
GPU 1: [a_sum, b_sum, c_sum, d_sum]
GPU 2: [a_sum, b_sum, c_sum, d_sum]
GPU 3: [a_sum, b_sum, c_sum, d_sum]

各自除以 N 得到平均值。
```

**Ring AllReduce 的优势**：通信量 ≈ 2 × 梯度大小（与卡数无关），带宽利用率高。

**通信量**：每次 backward 需要同步所有参数对应的梯度。对于 25M 参数模型（FP32），每次通信约 100MB。

**通信带宽**：
| 互联方式 | 带宽 | 适用场景 |
|---------|------|--------|
| NVLink | 900 GB/s（H20） | 同机多卡，首选 |
| PCIe Gen5 | ~64 GB/s | 同机多卡，无 NVLink |
| InfiniBand/RoCE | 200-400 Gbps | 跨机多卡 |

### 4.5 DP vs DDP 对比

| 维度 | DataParallel (DP) | DDP |
|------|-------------------|-----|
| 并行方式 | 多线程（GIL 限制） | 多进程（无 GIL） |
| GPU0 负载 | 不对称，更忙 | 对称，负载均衡 |
| 模型复制 | 每 step 复制一次 | 只在初始化时复制 |
| 梯度同步 | gather 到 GPU0 | all-reduce，所有卡参与 |
| 扩展性 | 2 卡还行，4 卡打折扣 | 线性扩展到 8 卡+ |
| 启动方式 | `python train.py` | `torchrun --nproc_per_node=N` |
| Windows 支持 | ✅ | ⚠️ 有兼容问题 |
| **推荐场景** | **快速实验、2卡以内** | **正式训练、2卡以上** |

## 4.6 DDP 常见问题与 NCCL 超时诊断

### 问题：DDP 训练时 NCCL Watchdog 超时

#### 现象

```
[rank1]:[E708 16:57:15.611356537 ProcessGroupNCCL.cpp:2241]
[PG ID 0 PG GUID 0(default_pg) Rank 1] failure detected by watchdog at work sequence id: 113281
PG status: last enqueued work: 113289, last completed work: 113280

Watchdog caught collective operation timeout:
WorkNCCL(SeqNum=113281, OpType=BROADCAST, ...) ran for 600091 milliseconds before timing out.
```

**特征**：
- 训练前几个 batch 正常，然后突然卡住
- 不是一开始就显存溢出（能跑好几个 batch）
- 多卡 DDP 训练才会出现（单卡无此问题）

---

#### 根本原因

**NCCL 的默认超时时间太短**：600 秒（10 分钟）

```
超时过程：
1. DDP backward 时自动触发 all-reduce
2. 梯度同步需要时间（特别是大模型 + 复杂通讯拓扑）
3. 某个 rank 由于各种原因稍微慢了点（显存竞争、调度延迟等）
4. 其他 rank 等待该 rank，超过 600 秒就认为死了
5. NCCL Watchdog 触发，强制关闭进程

潜在原因：
- ✅ 梯度计算量大，all-reduce 真的很慢（但不是我们的情况）
- ✅ batch_size 太大导致显存压力大，GC 暂停时间长
- ✅ 某卡上的计算任务多（CPU 被抢占）
- ✅ 网络拥塞导致集合通讯延迟
- ✅ find_unused_parameters=True 导致 DDP 每次都扫描未使用参数（浪费时间）
- ✅ static_graph=True 但计算图实际会变（DDP 重复初始化）
```

---

#### 解决方案

**1. 增加 NCCL 超时时间（快速修复）**

```python
import os

# 设置在 import torch 之前！
os.environ['NCCL_TIMEOUT'] = '1800'  # 30分钟，而不是默认的 600秒（10分钟）
os.environ['NCCL_BLOCKING_WAIT'] = '1'  # 强制同步，让 rank 间更严格的等待
os.environ['TORCH_NCCL_BLOCKING_WAIT'] = '1'  # PyTorch 层面的同步

import torch
import torch.distributed as dist
```

**超时值建议**：
- 小模型（7B）：600-1000 秒
- 中等模型（13B-70B）：1200-1800 秒  ← **推荐**
- 大模型（>100B）：3600+ 秒
- 跨节点大规模训练：3600-7200 秒

---

**2. 优化 DDP 参数（根本修复）**

```python
from torch.nn.parallel import DistributedDataParallel as DDP

# ✅ 关键：find_unused_parameters=False
# 这会让 DDP 跳过"找未使用参数"的扫描，节省大量时间
model = DDP(
    model,
    device_ids=[local_rank],
    gradient_as_bucket_view=True,  # 省一份梯度拷贝
    static_graph=True,  # 计算图固定（如果模型结构确实固定）
    find_unused_parameters=False  # ← 最重要！
)
```

**各参数说明**：

| 参数 | 值 | 作用 | 性能影响 |
|------|-----|------|---------|
| `find_unused_parameters` | False | 不扫描未用参数，加快初始化 | +5-10% |
| `gradient_as_bucket_view` | True | 梯度作为桶视图，省内存 | +内存 -时间 |
| `static_graph` | True | 固定计算图，加快 all-reduce | +10-20% |
| `broadcast_buffers` | True | 同步 BN running_mean 等 | +通讯 |

---

**3. 检查数据加载平衡**

```python
# ✅ 确保各卡数据大小一致
sampler = DistributedSampler(
    dataset,
    num_replicas=world_size,
    rank=rank,
    shuffle=True,
    drop_last=True  # ← 关键！丢弃末尾不足 batch 的数据
)
```

**为什么需要 drop_last=True**：
```
无 drop_last：
- rank0 最后拿到 batch_size=64
- rank1 最后拿到 batch_size=60（数据集大小不是整数倍）
→ all-reduce 时两卡张量大小不同，死锁！

使用 drop_last=True：
- 丢弃末尾不足 batch 的数据
- 所有 rank 的 batch_size 完全相同
→ 集合通讯正常
```

---

**4. 诊断超时的根本原因**

如果加大超时时间后仍然超时，说明问题在别处。启用 NCCL 跟踪：

```bash
# 启用 NCCL 调试日志
export TORCH_NCCL_TRACE_BUFFER_SIZE=10000
export NCCL_DEBUG=INFO

# 启动训练
torchrun --nproc_per_node=2 train.py 2>&1 | tee training.log

# 查看日志中的 NCCL 操作

```

**典型的诊断日志**：
```
[rank0] [0] NCCL INFO Channel 00 : 2[0] -> 2[1] via direct shared memory
[rank0] [0] NCCL INFO Using 1 threads, min block size 32768, max block size 16777216
[rank0] [0] NCCL INFO Trees [0] -1 -1, [1] 1 -1, [2] -1 1, [3] 1 -1
[rank0] NCCL operation broadcast timed out after 600.0 seconds
  ↑ 说明确实超时了，不是其他问题
```

#### 完整修复示例

```python
import os

# 设置在最开始
os.environ['PYTORCH_CUDA_ALLOC_CONF'] = 'expandable_segments:True'
os.environ['NCCL_TIMEOUT'] = '1800'  # 30分钟超时
os.environ['NCCL_BLOCKING_WAIT'] = '1'
os.environ['TORCH_NCCL_BLOCKING_WAIT'] = '1'

import torch
import torch.nn as nn
import torch.distributed as dist
from torch.nn.parallel import DistributedDataParallel as DDP
from torch.utils.data.distributed import DistributedSampler

def setup_ddp():
    dist.init_process_group(backend='nccl')
    local_rank = int(os.environ['LOCAL_RANK'])
    torch.cuda.set_device(local_rank)
    return local_rank

def train():
    local_rank = setup_ddp()
    rank = dist.get_rank()
    world_size = dist.get_world_size()
    device = torch.device(f'cuda:{local_rank}')

    # 模型
    model = YourModel()
    model.to(device)

    # ✅ DDP 包装（关键参数）
    model = DDP(
        model,
        device_ids=[local_rank],
        gradient_as_bucket_view=True,
        static_graph=True,
        find_unused_parameters=False  # ← 最重要
    )

    # 数据加载
    sampler = DistributedSampler(
        dataset,
        num_replicas=world_size,
        rank=rank,
        shuffle=True,
        drop_last=True  # ← 重要
    )
    dataloader = DataLoader(dataset, sampler=sampler, drop_last=True)

    optimizer = torch.optim.AdamW(model.parameters(), lr=1e-4)

    # 训练循环
    for epoch in range(50):
        sampler.set_epoch(epoch)  # ← 每 epoch 都要设置

        for batch_idx, (x, y) in enumerate(dataloader):
            x, y = x.to(device), y.to(device)

            # 前向传播
            logits = model(x)
            loss = nn.functional.cross_entropy(logits.view(-1, logits.size(-1)), y.view(-1))

            # 反向传播（DDP 在这里自动 all-reduce）
            optimizer.zero_grad()
            loss.backward()
            optimizer.step()

            if batch_idx % 100 == 0 and rank == 0:
                print(f"Epoch {epoch}, Step {batch_idx}, Loss: {loss.item():.4f}")

    dist.destroy_process_group()

if __name__ == '__main__':
    if 'LOCAL_RANK' not in os.environ:
        print("Must use torchrun to launch!")
        exit(1)

    train()
```

**启动命令**：
```bash
# 2 卡训练
torchrun --nproc_per_node=2 train.py

# 4 卡训练
torchrun --nproc_per_node=4 train.py
```

❌ **错误 1**：仅增加超时，不改 DDP 参数
```python
os.environ['NCCL_TIMEOUT'] = '3600'  # 这样只是掩盖问题
# ↑ 会延迟超时，但仍然是治标不治本
```

❌ **错误 2**：使用 synchronize() 调试
```python
# DON'T DO THIS
if batch_count % 100 == 0:
    torch.cuda.synchronize()  # 这会显著拖慢训练！
```

❌ **错误 3**：find_unused_parameters=True 但又不改模型
```python
# DON'T DO THIS

```

如果是多节点训练（节点间通过网络连接），还需要：

### 常见误区：Buffer 同步导致的 Epoch 级别超时

#### 现象（第2个 epoch 后超时）

```
第1个 epoch: ✅ 正常运行
第2个 epoch: ❌ 超时

错误信息：
[rank1]: File "...torch/nn/parallel/distributed.py", line 2190, in _sync_buffers
[rank1]:     self._sync_module_buffers(authoritative_rank)
[rank1]:   File "...torch/nn/parallel/distributed.py", line 2194, in _sync_module_buffers
[rank1]:     self._default_broadcast_coalesced(authoritative_rank)
[rank1]: torch.distributed.DistBackendError: Watchdog caught collective operation timeout:
          WorkNCCL(SeqNum=258244, OpType=BROADCAST, ...) ran for 600000 milliseconds before timing out.
```

**特征**：
- 第1个 epoch 能正常通过
- 第2个 epoch 开始时卡住
- 错误在 `_sync_buffers()` 中发生

#### 根本原因

**DDP 的 buffer 同步机制**：

在每个 epoch 开始（更准确地说，在每个 forward 前），DDP 会尝试同步所有 buffers（如 BatchNorm 的 `running_mean` 和 `running_var`），以保证各卡的这些统计量一致。这个同步操作是一个 **BROADCAST** 集合通讯。

```
Epoch 开始:
sampler.set_epoch(epoch)
    ↓
第一次 forward:
model(x)  ← DDP 在这里触发 _pre_forward
    ↓
_sync_buffers()  ← 尝试广播所有 buffers
    ↓
BROADCAST 超时 ← 某个 rank 响应太慢
```

**为什么第1个 epoch 不超时而第2个超时？**

```
第1个 epoch 开始:
- 模型刚初始化，buffer 都是默认值
- BROADCAST 数据量小（全是 0）
- 通讯快速完成

第2个 epoch 开始:
- Buffer 已经被更新（BN 在第1个 epoch 中累积了统计）
- 或者某个 rank 显存压力大、GC 暂停等原因响应慢
- BROADCAST 操作被阻塞
```

#### 解决方案：禁用 Buffer 同步

对于大多数训练场景，BN 的 running stats 同步不必要（特别是小模型），可以禁用：

```python
model = DDP(
    model,
    device_ids=[local_rank],
    find_unused_parameters=False,
    static_graph=True,
    broadcast_buffers=False  # ← 禁用 buffer 同步！
)
```

**各 DDP 参数详解**：

| 参数 | 值 | 说明 | 何时需要 |
|------|-----|------|---------|
| `find_unused_parameters` | False | 不扫描未用参数，加快初始化 | 模型中所有参数都会用到时 |
| `static_graph` | True | 固定计算图，加快 all-reduce | 模型结构不变（无条件分支） |
| `broadcast_buffers` | **False** | 禁用每个 forward 前的 buffer 同步 | **大多数场景都应该 False**（除非需要 BN 的跨卡统一） |
| `gradient_as_bucket_view` | True | 梯度作为桶视图，省内存拷贝 | 想节省显存时 |

**为什么大多数场景应该 False？**

```
broadcast_buffers=True 的代价：
1. 每个 forward 前都要同步 buffer → 额外通讯
2. Buffer 数据量虽小，但频繁的小通讯会引发同步问题
3. 对于小模型，BN 的 running stats 不重要

broadcast_buffers=False 的影响：
- BN 的 running_mean/running_var 不再跨卡同步
- 各卡的 BN 使用自己局部的统计值
- 但梯度仍然同步（backward 的 all-reduce）
- 推理时基本无影响（通常用 eval 模式，BN 不起作用）
- 训练时的影响很小（BN 本身就只是稳定化，不是核心）
```

#### 完整修复代码

```python

# ⚠️ 必须在 import torch 之前设置！

# ... 模型定义 ...

# ✅ DDP 包装：完整的最优参数
model = DDP(
    model,
    device_ids=[local_rank],
    # 性能优化
    find_unused_parameters=False,      # 不扫描未使用参数
    static_graph=True,                 # 固定计算图
    gradient_as_bucket_view=True,      # 省梯度拷贝
    # 稳定性优化

)

# ✅ DistributedSampler：确保各卡数据一致
sampler = DistributedSampler(
    dataset,
    num_replicas=world_size,
    rank=rank,
    shuffle=True,
    drop_last=True  # ← 重要！丢弃末尾不足 batch 的数据
)

dataloader = DataLoader(
    dataset,
    batch_size=batch_size_per_gpu,
    sampler=sampler,
    drop_last=True,
    num_workers=0,
    pin_memory=True
)

# 训练循环
for epoch in range(epochs):
    sampler.set_epoch(epoch)  # ← 每 epoch 必须调用

    for x, y in dataloader:
        x, y = x.to(device), y.to(device)

        logits = model(x)
        loss = criterion(logits, y)

        optimizer.zero_grad()
        loss.backward()        # ← 梯度 all-reduce 在这里发生
        optimizer.step()
```

## 五、Tensor Parallelism (TP) 与 Pipeline Parallelism (PP)

> 了解为主，小模型用不到。

### 5.1 Tensor Parallelism (TP)

将一个矩阵乘法的矩阵**按列或按行切分**到多张卡上：

```
                    GPU 0          GPU 1
输入 x          →   x          →   x
                  ↓                ↓
权重 W（切列）     W[:, :n/2]       W[:, n/2:]
                  ↓                ↓
部分输出          z1               z2
                  ↓                ↓
              all-gather 合并 → 完整输出 z
```

- **优点**：单层的计算被拆分，可以突破单卡显存/算力限制
- **缺点**：每层都需要 all-gather 通信，延迟敏感
- **适用**：同机 NVLink 互联的卡（通信带宽够）

### 5.2 Pipeline Parallelism (PP)

将模型的不同层分配到不同卡上，数据像流水线一样通过：

```
GPU 0 (Layer 0-5)    GPU 1 (Layer 6-10)
    micro_batch_0 ──→
    micro_batch_1 ──→     micro_batch_0 ──→
    micro_batch_2 ──→     micro_batch_1 ──→
                          micro_batch_2 ──→
```

- **优点**：跨卡通信量小（只传激活值），适合跨机
- **缺点**：pipeline bubble（部分卡在等数据），需要 micro-batch 来减少气泡
- **适用**：跨机训练、模型太大单卡放不下

## 六、实验观察：DataParallel 在 2×H20 上的表现

### 6.1 实验配置

| 项目 | 配置 |
|------|------|
| GPU | 2× NVIDIA H20 96GB |
| 互联 | NVLink（理论 900 GB/s 双向） |
| 模型 | mini_gpt（embed=256, heads=4, layers=10, ~25M） |
| 并行策略 | DataParallel |
| batch_size | 调大（显存利用率 ~70%） |
| 精度 | FP32 |

### 6.2 观察到的现象

**现象 1：NVLink 带宽 14.4 GB/s**

H20 NVLink 理论 900 GB/s，实际只用了 14.4 GB/s（利用率 1.6%）。

**原因**：模型参数（~100MB）和 batch 数据太小，DP 每 step 的通信量（scatter/gather/replicate/broadcast）加起来也就几百 MB，根本吃不满 NVLink。

**现象 2：Tensor Core 和 SM 利用率极低**

| 指标 | 观测值 | 说明 |
|------|--------|------|
| FP16 活跃度 | < 1% | FP32 训练，几乎没用 FP16 |
| FP32 活跃度 | < 3% | 模型太小，计算量不够 |
| Tensor Core | 12.7% | 矩阵太小，TC 启动开销 > 收益 |
| SM 占用率 | 30-40% | 大量 SM 在"空跑"或等数据 |

**原因分析**：

```
mini_gpt 每个 step 的计算量：
FLOPs ≈ 6 × 25M × (32 × 128) ≈ 600 GFLOPs

H20 FP16 算力 ~448 TFLOPS
利用率 ≈ 600G / 448T ≈ 0.13%
```

25M 参数的小模型在 H20（算力数百 TFLOPS）上严重"吃不饱"。SM 被调度了（30-40%）但实际计算量极少（FP16 <1%），大量时间在等 Python 循环、DataLoader、小矩阵调度。

**现象 3：两卡利用率不对称**

GPU0 的 SM/FP16/FP32 占用率明显高于 GPU1，但 Tensor Core 和 DMA 利用率基本持平。

**原因：DataParallel 的工作流天然不对称**

| 操作 | GPU0 | GPU1 |
|------|------|------|
| forward/backward | ✅ | ✅ |
| loss 计算 | ✅ | ❌ |
| gather 输出/梯度 | ✅（接收方） | 发送 |
| optimizer step | ✅ | ❌ |
| broadcast 参数 | ✅（发送方） | 接收 |

- GPU0 额外做了 loss 计算、梯度汇总、参数更新、参数广播，所以更忙
- Tensor Core 持平是因为 forward/backward 的矩阵乘法两卡一样
- DMA 持平是因为两卡都做数据搬运（scatter/gather）

### 6.3 结论

| 观察 | 是否合理 | 根因 |
|------|---------|------|
| NVLink 1.6% | ✅ 合理 | 模型/数据太小，通信量不够 |
| Tensor Core <1% FP16 | ✅ 合理 | FP32 训练 + 模型太小 |
| SM 30-40% | ✅ 合理 | 小矩阵调度，SM 空跑 |
| 两卡不对称 | ✅ 合理 | DataParallel 固有问题 |

**核心结论：25M 的小模型不适合多卡训练。** 多卡不但不能加速，反而增加了通信和同步开销。

## 七、加速建议决策树

```
训练速度慢
├── 显存不够？
│   ├── 单卡放得下 → 不需要多卡，优化 batch_size / AMP / gradient checkpointing
│   └── 单卡放不下 → 多卡数据并行 (DDP)
│       └── 模型太大单卡放不下 → 模型并行 (TP/PP)
│
├── 显存够但速度慢？
│   ├── GPU 利用率低（SM <50%）→ 瓶颈不在 GPU，先优化：
│   │   ├── DataLoader 瓶颈 → 增大 num_workers、prefetch
│   │   ├── Python 循环瓶颈 → 减少 Python 开销、用 torch.compile
│   │   └── 模型太小 → 增大模型/batch_size/seq_len
│   │
│   └── GPU 利用率高（SM >80%）→ 算力瓶颈，多卡有效：
│       ├── 2-4 卡 → DDP
│       └── 8 卡以上 → DDP + 模型并行
│
└── 模型规模选择
    ├── < 100M 参数 → 单卡足够，多卡收益极低
    ├── 100M - 1B → DDP 2-4 卡有效
    ├── 1B - 10B → DDP + TP
    └── > 10B → 3D 并行 (DP + TP + PP)
```

## 九、关键概念速查

| 概念 | 解释 |
|------|------|
| **all-reduce** | 分布式通信原语：所有进程的梯度求和取平均，结果分发给所有进程 |
| **scatter** | 将数据切分后分发到各 GPU |
| **gather** | 将各 GPU 的数据汇聚到一个 GPU |
| **broadcast** | 将一个 GPU 的数据广播到所有 GPU |
| **effective batch size** | 每卡 batch_size × 卡数，DDP 的总 batch |
| **NCCL** | NVIDIA Collective Communications Library，GPU 间集合通信库 |
| **torchrun** | PyTorch 分布式启动器，自动设置环境变量和进程组 |
| **LOCAL_RANK** | 当前进程在本机的 GPU 编号（0, 1, 2, ...） |
| **RANK** | 当前进程在全局的编号（跨机时 = 机器号 × 每卡数 + LOCAL_RANK） |
| **WORLD_SIZE** | 总进程数（= 机器数 × 每卡 GPU 数） |
| **pipeline bubble** | PP 中部分 GPU 空闲等待数据的时间浪费 |
| **gradient accumulation** | 累积多个 mini-batch 的梯度后再更新，等效于增大 batch_size |
