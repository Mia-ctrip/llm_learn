# 文档 2：GPU 训练平台技术架构介绍 - 面试问答档案

**GPU 训练平台 · 技术架构与实践**

面试问答档案 ｜ 基于多年 AI Infra 一线工程实践整理

**一、平台概述**

**1.1 平台定位**

面向算法工程师与数据工程师的一站式 GPU 训练平台，统一承载从模型开发、调试、分布式训练到任务编排、资源治理的全链路工作。平台核心目标是：

屏蔽底层基础设施差异（多云 / 私有云 / 多版本 Kubernetes），向上层提供稳定一致的算力交付能力。

以租户与配额为核心，支撑数十个业务团队、数千张 GPU 卡的资源共享与隔离。

通过精细化调度与治理，持续提升 GPU 利用率、降低单位算力成本。

**1.2 核心术语**

为后续讨论统一语义，平台定义了如下基础概念：

**• 租户（Tenant）：**资源分配的逻辑单元，按业务方维度划分，承担配额申领、计费归属、权限隔离三类职责。

**• 资源单元（Resource Unit）：**资源分配的最小粒度，由“卡型 / 卡数 / 显存”三元组确定，例如 L20/1/48 表示 1 张 L20 显卡、48G 显存。

**• GPU 配额（Quota）：**租户被授予的可用资源单元数量上限，开发机、自定义任务、推理服务统一从 Quota 中扣减。

**• 开发机（DevBox）：**常驻 Pod 形态的交互式环境，用于代码开发与调试。

**• 自定义任务（Job）：**基于稳定代码与镜像、可周期化或一次性提交的训练 / 数据处理任务，支持单机多卡与多机多卡分布式。

**二、整体技术架构**

**2.1 分层视图**

平台按职责自上而下划分为四层：

接入与门户层：Web 控制台、OpenAPI、CLI，承载用户交互与租户自助化操作。

平台业务层：Java / Spring Boot 实现，负责任务管理、资源管理、配额计费、监控治理等核心业务编排。

调度与编排层：基于 Kubernetes，自研 Operator 与 Webhook，扩展原生调度能力，支持 Gang 调度、自定义 BinPack、租户级 Best-Effort 等策略。

基础设施层：异构 GPU 节点（多卡型、多代际）、容器运行时、IB / RDMA 高性能网络、分布式存储（CephFS / JuiceFS / 块存储），通过 IaC 在多云环境下统一交付。

**2.2 平台业务层职责**

该层是平台的“大脑”，承载全部领域模型与业务规则，主要包含：

任务管理：任务创建、提交、状态跟踪、状态通知；统一抽象自定义任务、开发机、推理服务三类工作负载的生命周期。

资源管理：以租户为维度管理 GPU、CPU、内存、存储 Quota，支持申请、调整、退订、再分配；提供分时调度、混合调度等策略以最大化资源利用率。

计费与成本分摊：基于卡型单价、占用时长、租户归属进行成本核算，输出账单与用量报表。

监控与治理：实时采集实例 / 容器 / 利用率 / 池使用率等多维指标，驱动告警、自动关机、配额回收等治理动作。

权限与流程：基于租户 / 角色的访问控制，覆盖资源申请、审批、变更全流程审计。

**2.3 调度与编排层**

调度层在 Kubernetes 之上构建了三类核心组件：

**• API Gateway：**统一封装 Kubernetes 原生资源（Pod / Job / PVC / Event / Log）与自定义资源（CRD）的访问，向上屏蔽 K8s SDK 细节，向下解决多集群、多版本兼容问题。

**• Operator（Controller-Runtime）：**负责自定义资源的生命周期编排：StatefulSet / Service / PVC 等子资源的创建与回收、状态同步、事件回调，以及与服务网格（Gateway / VirtualService / DestinationRule）的集成。

**• GPU Webhook：**准入阶段对 Pod 进行变更：动态注入 GPU 资源声明（如 nvidia.com/gpu）、IB / RDMA 共享设备（rdma\_hca\_shared）、节点亲和与污点容忍，实现“用户不感知、平台强约束”。

**2.4 基础组件**

nvidia-device-plugin：节点 GPU 资源上报，是调度器进行卡数分配的事实来源。

dcgm-exporter：采集容器 / 宿主机粒度的 GPU 利用率、显存、温度、功耗等指标。

CSI 驱动 + VolumeSnapshot：为开发机一键转任务等场景提供 PVC 快照与克隆能力。

节点初始化（Image / 驱动 / 容器运行时）：通过标准化镜像（Alma Linux + 指定版本 NVIDIA Driver）保证多云节点的一致性。

**三、核心领域模块详解**

**3.1 任务管理**

任务管理模块是平台用户感知最强的部分，主要支撑两类典型工作负载：

开发机：常驻型 Pod，支持 SSH / 网页终端 / IDE 远程接入，挂载持久化卷与共享存储；提供启停、重建、规格调整、自动关机等运维能力。

自定义任务：以 Kubernetes Job / Gang Job 为底座，支持单机单卡到多机多卡的分布式训练；任务状态机覆盖 排队 → 调度中 → 运行 → 成功 / 失败 / 取消 全生命周期。

代码层面，相关编排逻辑由 SchedulerTaskBiz、AdhocTaskBiz、ApplyProcessBiz、JobDetailBiz 等模块组成，分别处理调度型任务、即席任务、申请审批流、任务详情与日志回放。

**3.2 资源管理**

资源管理由“资源池 - 租户 - 配额 - 实例”四级模型构成：

资源池（PoolResource）：将物理节点按卡型、用途（训练 / 开发 / 推理）切分为若干池，作为调度的最小可见单位。

租户视图（TenantResourceManager）：对每个租户聚合其在各资源池下的总量、已用、空闲、欠分四类指标，作为门户展示与告警依据。

资源申请（ResourceBiz / GpuApplyBiz）：覆盖新申请、扩容、缩容、跨租户调拨；写入流程同步与异步双通道，并与审批系统打通。

实例管理（ResourceInstanceBiz）：将一次具体的资源占用（一台开发机、一次任务运行）实例化，统一承担状态机、计费起止时间、归属租户等职责。

**3.3 GPU 配额申请与分配**

配额变更场景复杂多样，工程上抽象为策略模式，由 QuotaProcessorManager 统一调度：

NewApplyProcessor：新申请 / 扩容；进行配额合法性、库存可用性、上下限三类校验。

ReallocateProcessor：跨租户 / 跨池调拨；保证总账平衡，避免出现“双花”。

DefaultProcessor：兜底分支，统一异常处理与日志埋点。

每个 Processor 返回结构化结果（QuotaValidationResult / NewApplyValidationResult / ReallocateValidationResult），便于上游进行幂等重试与失败回滚。

**3.4 调度策略**

**3.4.1 自定义 BinPack — 减少资源碎片**

节点评分倾向：优先选择剩余 GPU 较少的节点，把零散资源“填满”，避免大任务被小任务卡住。

奇偶匹配优化：AI 任务多以 1 / 2 / 4 / 8 卡为单位，调度器优先调度到“可用 GPU 数为奇数”的节点，提升非 1 卡任务的成功率与整盘利用率。

**3.4.2 Gang 调度 — 多 Pod 成组拉起**

分布式训练要求所有 Worker 同时就绪，否则会出现“部分 Pod 占着卡空等”。Gang 调度借助调度器的成组语义，确保只有在最小成员数都可调度时才一起绑定节点，否则全体回退到队列。

**3.4.3 租户内 Best-Effort**

原始设计中“资源类型独占”会导致大规格资源（如 8 卡机）无法承载小规格任务，造成空载。Best-Effort 策略允许租户内不同规格之间相互填充，能跑尽跑，显著提升资源吞吐。

**3.4.4 任务排队与优先级**

放宽提交期校验，允许任务在租户内统一排队；调度器以 score = w\_p·priority + w\_a·wait\_time − w\_s·size\_penalty + w\_f·fairness\_boost 综合评分，兼顾优先级、公平性、防饿死、避免大任务长期挡队。

**3.5 计费与成本分摊**

计费模块（GpuCostBiz / GpuQuotaBiz / CostShareBiz）以“卡型单价 × 占用时长 × 租户归属”为基本公式，支持三类视角：

租户视角：日 / 月维度账单，按资源池与卡型展开，附带可下钻的实例明细。

业务方视角：将多个租户聚合到上层业务，用于成本归集与跨部门分摊。

平台视角：库存、空置、共享池缓冲等口径，支撑容量规划与采购决策。

为了避免账单与实际资源不一致，计费数据来源于资源实例的状态机事件（启动 / 终止 / 缩扩容），而非定时拉取的快照，保证账单可审计、可重算。

**3.6 监控体系**

监控分三个层次：

指标采集：dcgm-exporter / kubelet / 自研 Agent 上报容器与宿主机指标（GPU 利用率、显存、SM 占用、温度、功耗、磁盘 IO 等）。

指标聚合（GPUResourceMonitorBiz / MonitorBiz / QuotaMonitorV2Biz）：按实例、容器、租户、资源池四个维度聚合，落库后供报表与告警查询。

告警与可视化：对接公司级时序数据库与告警通道，提供利用率分布、空闲热点、配额预警等仪表盘。

监控不仅是“看图”，更是治理的输入：所有自动化治理策略（关机、回收、降级）都基于同一份指标口径，避免“监控与治理对不齐”的常见坑。

**四、关键技术专题**

**4.1 通用 Gang Job 集群拓扑注入**

**问题背景**

原方案依赖 Kubernetes v1.28 引入的 Job Completion Mode（completion-index 标识序号），将 index=0 的 Pod 作为 Master。但生产中存在多套低版本（v1.19）集群，没有该特性，分布式训练在这些集群上无法直接落地。

**整体方案**

为每个 Job 注入一个 Init Container 作为“栅栏”：在 Gang Scheduler 把所有 Pod 拉起的过程中，Init Container 持续监听这组 Pod 的状态，待数量达到最小副本数后，对 Pod 名（含 Cluster Domain）排序并写入 /etc/gang/hostfile。训练启动脚本据此约定 Master 与 Worker 角色，不再依赖 K8s 版本特性。

**关键工程点**

Init Container 等待存在卡死风险：必须设置超时与重试上限，超时后让该副本快速失败，避免拖垮整组。

Pod 排序需稳定：以 Pod 名 + UID 排序，保证多次重试得到同一份 hostfile。

与 Webhook 协同：通过 Webhook 注入 Init Container，业务镜像零侵入。

**4.2 GPU 虚拟化（Koordinator + HAMi / MIG）**

**问题**

原生 K8s 只能整卡分配，开发机 / 推理类负载长期把卡“占住但用不满”，常见利用率仅 10%~30%。

厂商生态差异大，不同 GPU/NPU 在驱动、调度上缺乏统一抽象。

**技术路线**

集群层：引入 Koordinator，扩展调度器以感知细粒度的 GPU SM / 显存维度，实现共享调度。

设备层：对性能敏感的卡（A100 / H100）使用 MIG 做硬件级切分；对开发机等性能不敏感场景使用 HAMi-core 做 CUDA API 层切分。

平台层：增加 vGPU 卡型（如 8c50g vGPU L20/1/24），用户无感切换；计费按虚拟规格比例核算。

**权衡**

HAMi 通过 CUDA API 劫持实现，性能损失约 10%~18%，开发机场景可接受，训练 / 推理生产环境慎用。

卡切分后 CPU 与 MEM 也需同比例切分，需要重新评估单 Pod 资源画像。

**4.3 开发机一键转自定义任务**

调研中观察到：用户在开发机上完成调试后，往往直接把训练跑在开发机上，导致开发机长期占用 8 卡资源、利用率却很低。一键转任务方案的目标是“在不破坏用户工作环境的前提下，把训练负载迁出开发机”。

**核心设计**

利用 CSI 的 VolumeSnapshot 能力，对开发机 PVC 做快照并克隆为新 PVC，挂载到 Job Pod，保证依赖环境与代码完全一致。

提交后开发机可降级为低规格（1 卡 / vGPU / 纯 CPU）甚至自动关机，Job 与开发机生命周期完全解耦。

多云适配：私有云依赖 Ceph，公有云依赖块存储（如 ESSD），上层抽象统一接口。

**4.4 调度优化路线图**

开发机 Descheduling：识别长期低利用率开发机，主动驱逐回租户共享池。

租户内 Best-Effort：放宽资源单元独占限制，已上线，显著缓解大规格机器空载。

任务排队队列：租户内统一排队 + 排队超时机制，避免任务“无声失败”。

共享资源池：将未分配库存、热备机器、临时回收的 Quota 纳入统一资源池，按使用量计费，给低频 / 评估类负载提供高性价比通道。

**五、GPU 资源治理实践**

**5.1 治理框架设计**

治理模块（biz/costsaving）采用“策略 + 执行器 + 通知”三段式架构：

CostSavingStrategy：策略接口，每条治理规则实现为独立策略（低利用率开发机、低利用率推理、空闲 Quota 等）。

CostSavingExecutor / GovernanceActionExecutor：统一驱动策略执行、收敛幂等性，并支持灰度发布与一键回滚。

ResourceNoticeService：将识别结果通过即时通讯渠道触达租户管理员与上级负责人，形成“识别 → 通知 → 自动动作 → 复核”闭环。

**5.2 主要治理动作**

**5.2.1 低利用率开发机告警 + 自动关机（LowGpuUtilizationStrategy）**

核心工作时段（10:00-18:00）：连续 4 小时 GPU 利用率 = 0 且无终端连接 → 通知用户 + 自动关机。

非核心时段（18:00-次日 10:00）：连续 2 小时 GPU 利用率 = 0 且无终端连接 → 通知用户 + 自动关机。

多重豁免：长跑任务、白名单用户、近期申请变更等场景跳过，避免误杀。

**5.2.2 低利用率推理服务治理（InferenceLowGpuUtilizationStrategy）**

近 7 日 GPU 利用率 = 0 → 暂停同类型新申请，提示该租户优先复用闲置实例。

使用周期内连续 7 日利用率为 0 → 通知缩容，缩容后若仍在原报备范围内可免审批扩容。

**5.2.3 空闲 Quota 定时回收（QuotaIdleStrategy / QuotaIdleBiz）**

按周维度产出租户级闲置报表，明确闲置卡数与对应预估成本，抄送租户管理员与上级负责人。

租户存在闲置 Quota 时，暂停同类型新申请，强制先“盘活”再新增，杜绝“申请惯性”。

使用周期到期自动缩减；若用户有继续使用需求，按既定流程申请延期。

**5.2.4 不按 Quota 值混合调度**

打破“资源类型独占”，允许大规格资源池承接小规格任务（如 8 卡机跑 1 / 2 卡任务）。

在调度准入侧加入 Best-Effort 校验，配合任务队列防止资源被低优任务长期霸占。

**5.2.5 申请门槛精细化**

申请流程新增必填字段：卡型、使用周期（起止）、预估周期内成本，强制使用方在申请前完成自我评估。

训练任务到期前消息提醒，未申请延期则按计划缩减；如需延期，由上级 / 风控部门审批，从机制上把控延期成本。

**5.3 治理成果**

全集群 GPU 利用率提升约 5 个百分点（部分核心池从 15% 提升至 20%）。

开发机日均 GPU 利用率从 8% 提升至 30%，P95 从 40% 提升至 100%；推理服务平均利用率从 10% 提升至 18%，P95 从 80% 提升至 90%。

全年节省 GPU 成本约 1500 万元，带动业务方主动参与“盘活闲置 + 精细申请”的治理文化。

**六、面试常见问答（Q&A）**

**6.1 系统设计与架构**

**Q：简要介绍一下你做的 GPU 训练平台的整体架构？**

平台分四层。最上层是接入与门户层，提供 Web 控制台、OpenAPI、CLI 给算法和数据工程师；其下是平台业务层，由 Java/Spring Boot 实现，承载任务管理、资源管理、配额计费、监控治理等领域模型；再往下是调度与编排层，构建在 Kubernetes 之上，通过 API Gateway、Operator、Webhook 三个核心组件扩展原生调度能力，支撑 Gang 调度、自定义 BinPack、Best-Effort 等策略；最底层是基础设施层，包括异构 GPU 节点、IB/RDMA 网络、分布式存储，跨多云通过 IaC 统一交付。整体目标是在保证多租户隔离的前提下，最大化资源利用率、降低单位算力成本。

**Q：为什么要在 Kubernetes 上自建一层平台，而不是直接让用户用 kubectl？**

Kubernetes 解决了容器编排，但不解决 AI 场景的若干关键问题：第一，用户大多是算法工程师，不应该让他们去维护 YAML 与 K8s 概念；第二，多租户配额、计费、审批属于业务能力，K8s 原生没有；第三，Gang 调度、GPU 拓扑、IB 注入等需要 Operator 与 Webhook 协同；第四，多集群、多版本（v1.19/v1.28）兼容必须靠平台抽象。我们在 K8s 之上做的恰恰是“面向算法工程师的领域抽象”。

**Q：租户 / 配额 / 资源池这套模型为什么这么设计？**

核心是把“分配语义”和“调度语义”解耦。资源池是物理可调度单元，按卡型、用途切分；租户是业务归属与计费单元；配额是租户在某资源池上的额度上限。三者解耦后，库存调整不会影响业务侧的可见配额，业务侧申请也不会直接绑死物理节点，调度器拥有最大化的灵活度。同时配额变更走策略模式（NewApply / Reallocate / Default），便于演进。

**6.2 调度与训练编排**

**Q：Gang 调度是什么？为什么 AI 训练必须用？**

Gang 调度是“成组调度”——一组 Pod 要么同时被调度，要么都不被调度。分布式训练里，所有 Worker 必须同时就绪才能开始 AllReduce 通信，否则先起来的 Pod 会占着 GPU 空等，既浪费又可能导致整体超时失败。Gang 调度通过最小成员数语义避免了这种部分启动场景。

**Q：你提到的“通用 Gang Job 集群拓扑注入”具体解决什么问题？**

原本的 Gang 实现依赖 K8s v1.28 的 Job Completion Mode，用 completion-index 来标识 Master。但我们的生产集群中存在 v1.19，没有这个特性，分布式训练无法直接跑。我们做的方案是给每个 Job 注入一个 Init Container，作为栅栏持续监听同组 Pod，待全部就绪后把排序后的 Pod 名写入 /etc/gang/hostfile，训练脚本读这个文件确定 Master / Worker。这样平台层就完全屏蔽了 K8s 版本差异。

**Q：自定义 BinPack 与 Kubernetes 默认调度有什么区别？**

默认调度器倾向 Spread，把 Pod 均匀打散，目的是高可用。但 GPU 训练场景下，Spread 会让每台机器都剩 1-2 张卡，结果一个 8 卡任务来时哪台都不够，整盘碎片化。自定义 BinPack 反其道而行：评分倾向选择剩余 GPU 较少的节点，把碎片节点先填满，给大任务留出整机。我们还做了一个奇偶匹配优化——AI 任务多是 1/2/4/8 卡，所以优先消耗“可用 GPU 为奇数”的节点，避免长期留下 1 张孤卡。

**Q：任务排队的优先级公式怎么设计？**

score = w\_p·priority + w\_a·wait\_time − w\_s·size\_penalty + w\_f·fairness\_boost。priority 是队列自带优先级，wait\_time 防饿死，size\_penalty 防止大任务一直占队头，fairness\_boost 给最近用得少的团队加分。权重通过线下回放历史排队数据来调，目标是“高优先任务首批进、长尾任务最终都能跑、单租户不会持续抢占”。

**6.3 资源治理与降本**

**Q：你做了哪些 GPU 治理动作？效果如何？**

核心三件事：第一，低利用率开发机告警 + 自动关机，按工作时段 / 非工作时段两套阈值识别空跑开发机；第二，空闲 Quota 定时回收，按周报表通知租户管理员，强制存在闲置时暂停同类型新申请；第三，不按 Quota 值混合调度，打破资源类型独占，让大规格资源池承接小任务。配套还有申请门槛升级、低利用率推理服务治理。整体效果是全集群 GPU 利用率提升约 5 个点，开发机日均利用率从 8% 提到 30%，全年节省约 1500 万元成本。

**Q：自动关机怎么避免误杀？**

多重保护：第一，识别口径不只是 GPU 利用率，还需要叠加“无终端连接”这个条件，因为有些用户在做数据预处理；第二，分时段阈值，工作时间放宽到 4 小时连续为 0，非工作时间收紧到 2 小时；第三，关机前先发通知，给用户主动续期或挂任务的机会；第四，白名单与豁免规则可配置，长跑任务、特殊业务可豁免；第五，所有动作可回放、可审计，出现误杀能复盘并快速回滚策略。

**Q：为什么先做治理而不是先做 GPU 虚拟化？**

治理收益边际更高、风险更低。治理是规则与流程层面的优化，几乎零性能损耗、可灰度可回滚；虚拟化（特别是 HAMi）会带来 10-18% 的性能损失，对训练 / 推理生产环境是硬约束。我们先用治理把“浪费”这部分挤干（约占 5 个点利用率），再用虚拟化解决“分配粒度太粗”这种结构性问题，路径上更稳妥。

**6.4 K8s 与基础设施**

**Q：Operator 在你们平台里承担了什么？**

Operator 基于 Controller-Runtime 实现，承担两类职责：一是自定义资源（CRD）的生命周期管理，把平台业务概念（开发机、训练任务等）下沉到 K8s 原语；二是子资源编排，比如自动创建 / 回收 StatefulSet、Service、PVC，并把状态同步回平台。这样平台业务层只需要关心“我要一个开发机”，剩下所有 K8s 细节由 Operator 闭环处理。

**Q：GPU Webhook 主要做了哪些事？**

做 Pod 创建准入阶段的变更：动态注入 GPU 资源声明（nvidia.com/gpu）、IB / RDMA 共享设备资源、节点亲和与污点容忍。这样用户提交任务时不需要写 GPU、IB 相关的字段，平台根据租户与资源单元统一注入，既保证一致性又便于策略演进。

**Q：多云 / 多集群是怎么统一管理的？**

节点交付层用 IaC（Terraform 等）统一节点规格、镜像、驱动版本（例如统一 Alma Linux + 指定版本 NVIDIA Driver）；K8s 层通过 API Gateway 屏蔽集群差异，平台业务层维护“逻辑集群 - 物理集群”映射；调度层在不同 K8s 版本上通过自研方案（如通用 Gang）兜底特性差异。整体目标是用户和上层业务感知不到底层在多云、多集群之间漂移。

**6.5 难点与亮点**

**Q：做这个平台你认为最有挑战的一块？**

是“治理与体验的平衡”。GPU 太贵，平台必须主动回收；但用户体感强，关错一台开发机可能影响一整天的工作。我们做了三件事来平衡：第一，所有自动动作都基于统一指标口径，并且监控 / 治理使用同一份数据，避免“监控显示忙、治理却以为闲”；第二，治理动作分级，从“通知 → 限制申请 → 缩容 → 关机”逐级升级，给用户预警与申诉时间窗口；第三，所有动作可灰度、可回滚、可审计，出现争议能复盘到具体策略与具体数据点。

**Q：如果让你重新设计这套平台，你会有哪些改进？**

第一，把调度策略做成“可插拔的策略中心”，让 BinPack / 奇偶匹配 / Best-Effort 等都成为可配置组合，便于实验与 A/B；第二，引入共享资源池 + 按使用量计费，把闲置 Quota、热备机器一并盘活，给低频任务一个高性价比通道；第三，监控、治理、调度共用一份指标真值（OLAP 化），杜绝口径漂移；第四，在性能不敏感的场景更激进地铺 GPU 虚拟化，把分配粒度从“整卡”降到“显存 + SM 配额”级别。

**附：自我对话清单**

面试时建议从“业务背景 → 架构分层 → 模块职责 → 关键专题 → 治理成果 → 反思迭代”六步推进，每一步用 30-60 秒覆盖核心要点，遇到深问随时下钻到具体模块（QuotaProcessor / Gang Init Container / LowGpuUtilizationStrategy 等）。控制总时长在 5 分钟内完成自述，把更多时间留给互动追问。
