# Skin Care Agent 前端设计与实现工作流

记录日期：2026-07-20

> **进度校准**：客户端已确定使用 React Native + Expo SDK 57，工程骨架已经创建；账号与首次协议闭环已有静态实现，正在等待真实前后端联调。本文后续步骤继续作为设计与实现顺序参考。

## 目标

以 Figma 作为设计真源，先建立可验证的视觉语言和组件体系，再实现 App 客户端，最终跑通：

`注册/登录 → 协议确认 → check-in → 三视角拍照 → AI 分析 → 日记 → 聚合结果 → 趋势/生命周期`

本流程规定设计与实现顺序；客户端技术栈现已确定为 React Native + Expo SDK 57。

## 核心原则

- 产品气质应冷静、可信、温和，不制造皮肤焦虑，也不呈现为医疗诊断工具。
- 用户照片和长期变化是主信息，装饰、动效和品牌表达不能干扰前后对比。
- 先确定视觉主张、内容层级和交互逻辑，再开始搭组件或写页面。
- Figma 中的变量、组件、状态和布局规则是设计真源；代码侧通过 Design-to-Code 和 Code Connect 尽量复用真实组件。
- 严重度和趋势不能只依赖红绿颜色表达；同时使用文字、图标或形状，兼顾色觉差异。
- 每个阶段都应有可检查产物和验收条件，未验证不得进入“已完成”。

## 工具组合

| 工具 / Skill | 用途 | 使用阶段 |
|---|---|---|
| `superpowers:brainstorming` | 明确用户目标、视觉方向、边界和成功标准 | 每个新设计模块开始前 |
| Figma MCP + `figma-generate-design` | 在 Figma 中创建完整页面和流程 | 页面设计阶段 |
| `figma-generate-library` + `figma-use` | 建立变量、Token、组件、Variants 和 Auto Layout | 设计系统阶段 |
| `figma-design-to-code` | 从已确认的 Figma 设计生成前端实现上下文 | 代码实现阶段 |
| `figma-code-connect` | 映射 Figma 组件与真实代码组件 | 组件体系稳定后 |
| `imagegen` | 情绪板、品牌插画、空状态素材和摄影方向探索 | 视觉探索阶段 |
| Expo 文档与开发工具 | 获取 Expo SDK 57 上下文、运行诊断、测试和调试 | 客户端实现阶段 |
| Storybook MCP（可选） | 读取和验证 React 组件状态与文档 | 选择 Expo/React 且组件体系稳定后 |

> `frontend-skill` 暂不列为已安装依赖。2026-07-20 实时检查 OpenAI curated 清单及 Git 仓库时未找到该 Skill；只有搜索缓存仍能访问旧内容。后续仅在官方仓库恢复可安装版本后再引入。

## 分阶段实施

### 1. 确定客户端技术栈（已完成）

已选择 React Native + Expo SDK 57，覆盖 iOS 和 Android。现有工程已经接入 Expo Router、Camera、Secure Store 和客户端环境配置；后续继续按 Expo 技术栈实现相机、图片处理、弱网重试、图表、动效、无障碍和自动化测试。

**产物**：技术选型已经落地为 `mobile/` Expo 工程。

### 2. 定义产品视觉方向

结合现有市场配色调研形成 2～3 个方向，再选择一个作为 MVP 主方向。每个方向明确视觉主张、色彩系统、字体与空间、摄影或插画风格、动效节奏，以及应避免的医疗冷硬感、低龄感和“通用 AI 卡片风”。

**产物**：情绪板、色板候选、关键页面低保真构图和评审结论。

### 3. 在 Figma 建立最小设计系统

- 建立颜色、字体、间距、栅格、安全区和页面边距变量；
- 创建按钮、输入框、选择器、标签、照片槽位、提示条、弹窗和底部导航；
- 覆盖默认、按下、禁用、加载、错误、空状态和权限拒绝；
- 建立皮肤指数、严重度、趋势曲线、区域生命周期和照片对比组件；
- 所有复用组件使用 Variants、Auto Layout 和语义化命名。

**产物**：Figma variables、基础组件库、组件状态页和使用说明。

### 4. 优先设计四个关键页面

1. 注册/登录及首次协议确认；
2. 三视角拍照与质量反馈；
3. AI 分析聚合结果；
4. 趋势和照片对比。

四个页面通过后，再补首页、日记、历史记录、生命周期详情、设置和账号注销。

**产物**：高保真关键页面、状态变体和可点击原型。

### 5. 做设计评审和小范围验证

检查非医疗定位是否清楚、拍照流程是否易懂、趋势是否避免放大单日焦虑、日记负担是否可接受，以及颜色对比、动态字体、触控区域、读屏标签和不同肤色下的可读性。优先使用 5～8 名目标用户做任务测试。

**产物**：评审清单、用户测试记录、修改结论和最终视觉方向。

### 6. 固化项目专属设计 Skill

视觉方向通过评审后，再创建 `skin-care-app-design` Skill，固化品牌气质与反例、设计 Token、组件规则、合规文案、照片和趋势呈现、无障碍约束，以及 Figma 到代码的验收清单。

不要在视觉方向尚未验证时创建，以免把临时偏好固化成长期规则。

### 7. 客户端基础骨架与账号链路（进行中）

- 已建立路由、主题、环境配置、API Client 和基础组件目录；
- Access Token 保存在内存，Refresh Token 使用系统安全存储；
- 已实现 Token 自动刷新、会话失效、统一错误处理、登录、注册、首次协议和占位首页；
- TypeScript 类型检查、ESLint 无缓存检查和 `git diff --check` 已通过；
- 尚待真实设备或模拟器上的前后端联调，以及单元测试和最小 UI 测试。

**第一条可验证链路**：注册、登录、首次协议确认、刷新 Token 和登出；当前实现已具备，运行时联调待完成。

### 8. 分三段跑通业务闭环

#### 8.1 拍摄与上传

实现 check-in 创建、front/left/right 三视角引导、质量反馈、上传进度、失败重试和幂等请求。

#### 8.2 分析与日记

实现分析等待状态、失败恢复、日记表单、聚合结果及就医提示；文案只描述外观状态。

#### 8.3 趋势与生命周期

实现趋势曲线、照片对比、partial/incomplete/superseded/legacy 状态，以及“仍可见、一次未见、连续未见”的生命周期表达。

每段完成后独立验证，不等整个 App 完成后一次性联调。

### 9. 建立 Figma 到代码的闭环

- 使用 Design-to-Code 获取布局、变量和组件上下文；
- 优先复用现有代码组件，不为每个页面复制新组件；
- 组件稳定后维护 Code Connect 映射；
- 对 Figma 设计稿和真机截图做逐屏对照；
- 设计变更先更新设计真源，再同步代码和组件文档。

### 10. 验证与内测准备

- 静态检查、单元测试和关键 UI 流程测试通过；
- 至少在一台 iPhone 和一台 Android 真机验证；
- 检查小屏、大字体、深色模式策略、权限拒绝、断网、Token 过期和分析失败；
- 验证照片隐私、协议门槛、账号注销和本地缓存清理；
- 形成 20～50 人内测包和问题反馈入口。

## 推荐执行顺序

1. 技术栈选型（已完成：React Native + Expo SDK 57）；
2. 市场调研分析与视觉方向；
3. Figma 最小设计系统；
4. 四个关键页面及原型；
5. 设计评审和目标用户验证；
6. 项目专属设计 Skill；
7. 客户端基础骨架与登录链路（进行中）；
8. 拍摄、分析、日记、趋势的分段实现；
9. Figma/代码一致性检查；
10. 真机验证和内测准备。
