# MVP 平台策略：App 还是微信小程序

> 决策日期：2026-07-16
> 当前阶段：后端 MVP 主链路已完成，下一步进入真实用户可操作闭环。

## 结论

1. MVP 阶段建议直接做 App，但只做可验证的窄 App。
2. 第一次不要先做小程序再迁移 App；建议直接做 App 版本，先小范围内测，再公开上架。
3. 小程序可以作为后续获客、传播和低门槛体验入口，但不作为第一阶段主产品。

## 为什么当前更适合 App

产品核心使用场景是：

```text
打开应用
→ 拍摄正面/左侧/右侧
→ 参考框辅助对齐
→ 等待分析
→ 查看趋势
→ 填写日记
→ 持续重复 30 天
```

这不是一次性工具，而是一个高频、长期、依赖相机体验和数据沉淀的产品。App 更适合以下能力：

- 相机权限、参考框、姿态提示和拍摄流程；
- 本地缓存、通知提醒、后台任务和图片处理；
- “这是我的皮肤记录工具”的产品心智；
- UI 动画、页面转场和视觉品质；
- 长期账号、照片和趋势数据沉淀。

本项目的 MVP 目标不是验证用户会不会点开，而是验证用户是否愿意连续记录 7～30 天。因此 App 更匹配当前验证目标。

## 小程序和 App 的取舍

| 目标 | 更适合的平台 |
|---|---|
| 快速验证用户是否愿意试一次 | 小程序 |
| 验证用户是否愿意连续记录 7～30 天 | App |
| 打磨拍照、参考框和视觉体验 | App |
| 微信社交传播 | 小程序 |
| 长期账号、通知、订阅和品牌心智 | App |

小程序的优势是无需安装、首次体验成本低、适合熟人传播。但本项目最重要的验证指标是持续记录和留存，而不是单次访问。

App 不天然等于更容易变现。iOS 数字内容和订阅需要考虑 App Store 规则与平台分成；符合条件的小型开发者可以申请 15% 费率，但仍有支付、审核和订阅设计成本。

参考：

- [Apple App Store Small Business Program](https://developer.apple.com/app-store/small-business-program/)

## 第一次上线的建议

建议直接做 App，但不要第一天就追求正式公开发布。推荐路径：

```text
App 内测版
→ 20～50 名真实用户
→ 验证 7～14 天使用行为
→ 修复拍照和记录流程
→ 小范围公开上架
→ 再扩大用户规模
```

因此不建议采用“小程序 MVP → 以后迁移 App”，而建议采用：

```text
App MVP → 内测 → 正式上线
```

小程序可以在后续作为轻量入口，但不应导致两套独立业务逻辑和数据体系。

## 如何避免未来平台迁移

迁移风险主要取决于账号和数据架构，而不只是客户端平台。现在就应保持：

- 后端用户 ID 不绑定 `wx_openid`；
- App 使用手机号、邮箱或其他独立登录方式；
- 微信登录只是可选身份来源；
- 照片、check-in、日记、分析和趋势都绑定后端用户 ID；
- App、小程序和 Web 未来共用同一套 API 和数据。

这样以后增加小程序，不是迁移用户数据，而是增加一个客户端。

## 上线前的额外风险

本项目涉及皮肤照片、健康相关信息和 AI 分析，App 上线准备会比普通工具严格。

Apple 对医疗和健康类 App 会关注准确性、免责声明、数据用途和隐私；如果产品声称诊断或治疗，审核会更加严格。产品应始终坚持“外观状态分析，不构成医疗诊断”的定位，并在严重情况时提示用户就医。

需要提前准备：

- 隐私政策和用户协议；
- 相机、照片和健康相关数据的授权说明；
- 用户数据删除和账号注销；
- AI 数据是否发送给第三方模型提供商的说明；
- 严重情况就医提示；
- App 备案、应用商店开发者账号和审核材料；
- 测试账号、审核说明和完整可运行的演示数据。

参考：

- [Apple App Review Guidelines](https://developer.apple.com/app-store/review/guidelines/)
- [Apple Health and Fitness Guidelines](https://developer.apple.com/health-fitness/)
- [Apple App Review：提交前常见问题](https://developer.apple.com/app-store/review/)
- [工信部：移动互联网应用程序备案工作说明](https://www.miit.gov.cn/jgsj/xgj/hlwgl/art/2023/art_564bf0759d7e41d5b4aa8ce4996b9e84.html)

## 第一版 App 的范围

第一版只完成一条可重复的主流程：

```text
注册/登录
→ 首次免责声明
→ 三视角拍照
→ 拍照质量提示
→ AI 分析
→ 查看本次结果
→ 填写日记
→ 查看趋势
```

暂时不做：

- 复杂社交；
- 多种账号体系；
- 护肤商城；
- 高级订阅权益；
- 过多算法指标；
- App 和小程序两套独立前端。

客户端已经选择 React Native + Expo SDK 57，覆盖 iOS 和 Android；后端继续复用当前 FastAPI API。第一阶段的目标是让约 20 名真实用户连续完成 7 天记录，而不是先做成完整商业产品。

## 最终决策

> 直接做 App MVP，先内测后正式上架；小程序作为未来的传播和低门槛入口，而不是第一阶段主产品。

## 2026-07-16 后端实施状态

App 路线所需的第一阶段后端基础化已经完成：

- 业务 API 统一迁移到 `/api/v1`；
- 移除固定开发用户，所有照片、check-in、分析、趋势、问答和生命周期接口均从 Bearer Token 获取当前用户并校验资源归属；
- `users` 只作为平台无关主体，原 `wx_openid` 已迁入 `user_identities(provider, provider_subject)`；
- 内测阶段先启用邮箱 + 密码，使用可撤销的 Access/Refresh Token，刷新时轮换两种 Token；
- 增加用户协议、隐私政策、健康免责声明和 AI 数据处理说明的版本化确认；未接受时业务接口返回 403；
- 增加账号注销与业务数据级联删除，并同步清理原图和标准化照片；
- check-in 和照片上传支持用户级 `client_request_id` UUID 幂等，适配移动网络重试；
- 新增迁移 `0012_app_foundation`，真实双用户链路已验证跨用户资源统一返回 404。

这套邮箱密码登录定位为 20～50 人内部测试方案。目前尚未实现邮件验证、找回密码、登录防爆破、生产对象存储和公开注册风控，因此不应直接作为公开上架版本。

## 2026-07-20 客户端实施状态

- React Native + Expo SDK 57 已确定为客户端技术栈，Expo 工程骨架已经创建。
- 当前客户端已实现登录、注册、Refresh Token 安全存储与轮换、首次四项协议确认、受保护路由和占位首页。
- TypeScript 类型检查、ESLint 无缓存检查和 `git diff --check` 已通过。
- 尚未完成真实设备或模拟器上的前后端端到端联调，因此账号闭环仍为进行中。
- check-in、三视角拍照、质量反馈、AI 分析、日记、趋势和生命周期页面尚未接入。

接下来按以下顺序联调和实现：

```text
注册/登录
→ 首次协议
→ 创建 check-in
→ 三视角拍摄与质量反馈
→ AI 分析与聚合结果
→ 日记
→ 趋势与生命周期
```
