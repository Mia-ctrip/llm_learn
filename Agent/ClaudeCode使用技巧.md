# Claude Code 使用技巧

> 课程笔记整理：涵盖 Claude Code 的核心命令、项目工作流、MCP、Subagent、Hooks、日志诊断和 Agent SDK。

---

## 一、常用斜杠命令（Slash Commands）

斜杠命令是 Claude Code 的核心交互方式，在终端中直接输入即可使用。

| 命令 | 说明 |
|------|------|
| `/content` | 查看当前上下文窗口包含的所有信息（加载的文件、系统提示词等） |
| `/status` | 查看当前配置状态（模型版本、上下文使用量、工具权限等） |
| `/init` | 以交互方式逐步生成项目的 `CLAUDE.md` 配置文件 |
| `/cd <path>` | 跳转到指定目录，改变工作路径 |
| `/compact` | 压缩上下文：保留摘要、丢弃细节噪声。消耗 token，耗时较长，适合**长任务的自然断点**处使用 |
| `/diff` | 查看当前分支的代码变更 |
| `/effort <level>` | 设置推理深度。**不是越高越好**，应根据任务难度选择合适级别，过高会浪费 token |
| `/verify` | 在本地浏览器中测试交付成果，检查 diff 文件变更行为是否符合预期 |

### 小技巧：环境变量优化 token 消耗

合理配置环境变量，可以显著减少每次请求携带的前置提示词（prefix prompt）长度，从而降低单次请求的 token 成本。

## 二、CLAUDE.md：项目核心配置文件

`CLAUDE.md` 是 Claude Code 的"项目说明书"，放在项目根目录，每次启动时自动加载进上下文。

- 通过 `/init` 命令以交互方式生成初始版本
- 典型内容：项目架构概述、技术栈、编码规范、常用命令、禁止操作
- 可结合社区插件（如 `memory.md`）让 Claude 在对话中主动将新学到的项目知识**沉淀回写**进配置文件，实现知识的持续累积

> `CLAUDE.md` 写得越清晰，Claude 的上下文理解越准确，产生幻觉的概率越低。

## 三、项目工作流

### 3.1 写新项目：先建立项目地图

在开始写代码前，先让 Claude 建立项目的整体认知地图（architecture map），明确：

- 目录结构与各模块的职责边界
- 模块间的依赖关系
- 技术选型与关键约束

这一步能避免 Claude 在不理解全局的情况下写出割裂的代码，也为后续的任务拆解提供上下文基础。

### 3.2 读已有项目

Claude Code 通过 **document index（文档索引）** 来检索和理解项目代码（类似 Codex 的工作方式）。

阅读项目的建议策略：

1. 先让 Claude 扫描目录结构，建立整体印象
2. 用提问代替直接修改："这个功能在哪里实现的？" 比 "直接改这个功能" 更安全
3. 大型项目可分模块逐步加载，避免一次性填满上下文

### 3.3 写新功能：结构化需求描述（mini-PRD 框架）

在向 Claude 下达新功能需求前，需明确以下四个维度，构成一份 mini-PRD：

| 维度 | 说明 |
|------|------|
| **验收标准** | 必须满足什么条件才算完成，输出可量化的 checklist |
| **非目标（Non-Goals）** | 明确这次**不做**什么，防止范围蔓延（scope creep） |
| **边界条件** | 缺失值、空值、异常情况如何处理 |
| **实施节奏** | 先评估影响面（哪些文件会被改动），人工确认后再批准执行 |

> 这个框架能有效减少 Claude 的"自由发挥"，让交付结果可预期、可验收。

## 四、MCP（Model Context Protocol）

MCP 是一种**标准化协议**，允许 Claude 连接并操作外部系统，突破纯代码编辑的局限。

常见的 MCP 接入场景：

- **数据库**：直接查询或写入数据
- **浏览器**：自动化操作网页
- **第三方 API**：调用外部服务
- **文件系统/代码执行环境**：在沙箱中运行代码并获取结果

通过 MCP，Claude Code 从一个代码编辑助手，升级为能与真实环境双向交互的智能体。MCP 服务的配置通常在 `CLAUDE.md` 或专用的 `.mcp.json` 配置文件中声明。

## 五、Subagent：上下文隔离与幻觉风险控制

### 核心理念

Subagent 是主 Agent 派生出的独立子任务执行者，核心价值在于**上下文隔离**——子任务在独立的上下文中运行，不污染主 Agent 的状态。

### 正确用法：只读 + Reviewer 角色

- **好的 Subagent**：只读不修改，用于检查、评估、审查
- **职责分离原则**：Subagent 的作用是**评估主 Agent 做的工作**，而不是和主 Agent 一起完成同一个功能

典型场景：主 Agent 实现代码变更 → Subagent 作为独立 reviewer 做 diff review 和逻辑校验 → 降低幻觉带来的 side effect。

### 常见错误用法与风险

| 错误用法 | 风险 |
|----------|------|
| 主任务不清晰时就启动 subagent | 双方上下文混乱，输出互相冲突 |
| 多个 agent 同时操作同一文件 | 产生竞争条件，内容互相覆盖 |
| 让 subagent 扫描全项目代码 | token 消耗剧增，性价比极低 |

## 六、Hooks：框架级自动化控制

Hooks 是**框架层面的能力，不是模型能力**——它在模型接收输入之前或之后介入，属于工程控制手段，不依赖模型的"理解"。

### 典型 Hook：`userPromptSubmit`

在用户提交 prompt **之前**触发，可以：

- 检测提示词是否存在歧义，有则先澄清再发给模型
- 自动注入项目上下文或规范约束
- 拦截潜在的危险指令

**执行流程**：用户输入 → `userPromptSubmit` Hook（评估/改写） → 模型处理

### 匹配方式（Matcher）

Hooks 提供两种 matcher 来精确控制触发时机，可按命令类型、文件类型、操作类型等维度进行匹配。

### 相关资源

- **`oh my claudecode`** 项目：开源的全自动智能体编排集合，提供大量现成的 hooks 示例，适合作为学习参考
- **`skill.md`** 文件：支持添加 `hooks` 属性，让技能文件具备自动触发能力，实现"技能 + 自动化"的组合

## 七、日志分析：排查问题的三个核心问题

当 Claude Code 的输出出现问题时，日志分析是定位根因的关键手段。

**排查时先问三个问题：**

1. **为什么贵**：哪些操作消耗了大量 token？是重复加载、无效工具调用还是 prompt 设计问题？
2. **为什么乱**：哪些行为偏离了预期轨道？是上下文污染还是指令歧义？
3. **为什么不及预期**：模型的思考路径是否走偏？最终输出在哪个节点开始偏差？

### 工具：`claude-code-log`

通过 `claude-code-log` 工具可以查看：

- 所有执行行为的完整记录（工具调用、文件读写等）
- 模型的 **thinking（思考过程）**，便于理解决策路径
- 识别偏离预期的具体步骤，用于复盘和调优

## 八、工具学习：通过 SDD 项目驱动实践

建议通过真实项目（如 SDD 项目）来系统学习 Claude Code 的各种工具能力。

**项目驱动学习的优势**：

- 在真实场景中感受工具的适用边界
- 通过实际报错和反馈，加深对 hooks、subagent 等机制的理解
- 比单纯阅读文档更高效，产出物可直接复用

## 九、Agent SDK：编程化构建多 Agent 系统

Agent SDK 是 Anthropic 提供的编程接口，允许开发者以**纯代码方式**构建和编排多 Agent 工作流，适合需要高度定制化的场景。

核心能力：

- 以代码定义 Agent 的工具集、系统提示和行为边界
- 编排多个 Agent 之间的协作关系（串行、并行、嵌套）
- 与现有系统深度集成（CI/CD、自动化测试、数据管道等）

与 Claude Code 交互式使用的区别在于：SDK 适合构建**无人值守的自动化流水线**，而 Claude Code 更适合**人机协作的开发场景**。
