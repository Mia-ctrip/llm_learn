## 1. 核心原则（这些原则决定所有取舍）

| 原则 | 含义 | 反面行为（不要做） |
|---|---|---|
| **训练为本，推理为翼** | 简历 / 项目 / 投递 70% 围绕训练平台，30% 围绕推理与 Agent | 把自己包装成"推理引擎专家"——你拼不过专业选手 |
| **一个深项目 > 三个浅项目** | 周末只做一个项目，分阶段递进 | 同时铺 RAG / Agent / 推理引擎 / 训练框架 |
| **学习服务于面试，而不是知识体系** | 每一项学习都要能产出"简历可写 + 面试可讲" | 为了知识完整去啃 Megatron / CUDA Kernel |
| **本职故事比新技能更重要** | ai-octopus 这个 3 年项目要打磨到 40 分钟不重复 | 急着学新东西，忽略最大杠杆 |
| **降本治理是你的招牌** | 1500 万降本 + Quota 治理 + 利用率提升要反复强化 | 让面试官把你和"普通 K8s 工程师"画等号 |

## 4. 周末核心项目：`hermes-serving` 训推一体化平台

**定位：** 不是"推理网关项目"，是"**训练平台的推理侧延伸**"——故事和你本职打通。

### 项目愿景（README 第一段就这么写）

> 复用携程 GPU 训练平台 3 年沉淀的调度治理思路，构建端到端训推一体化平台。
> 覆盖训练任务编排 → LoRA 微调 → 模型注册 → 推理服务部署 → KV Cache 治理 → 监控告警全链路。

### 三阶段递进

| 阶段 | 时间 | 核心交付 | 关联简历卖点 |
|---|---|---|---|
| **阶段 1**（M1-M2） | vLLM 部署 + OpenAI 兼容网关 + Prometheus 监控 + 推理服务自动注册 | "复用训练平台实例管理思路构建推理服务编排" |
| **阶段 2**（M3） | LoRA 微调链路 + 适配器注册 + 推理上线自动化 | "实现训练 → 微调 → 推理闭环，端到端部署 < 15 分钟" |
| **阶段 3**（可选 M4 行有余力） | 利用率治理策略（低 QPS 自动缩容、KV Cache 命中分析） | "把训练平台 1500 万降本的治理思路迁移到推理侧" |

### 技术栈（轻量、能跑、可讲）

- **推理**：vLLM（一个引擎吃透，不学 SGLang）
- **网关**：FastAPI（Python 栈与本职 Java 互补）
- **微调**：PEFT + LoRA（DeepSpeed 不学，太重）
- **监控**：Prometheus + Grafana + DCGM Exporter（**复用你训练平台经验**）
- **部署**：Docker Compose（够用，K8s Helm 可选）
- **IaC**：Terraform GPU 节点模块（**复用 packon 经验**）

### 不做的事（明确边界）

- ❌ 不做 SGLang / TGI 对比测试（深度不够，反而暴露）
- ❌ 不做 CUDA Kernel 优化
- ❌ 不做完整 Agent 平台（合并为"训练任务排障助手"小功能即可）
- ❌ 不做多模态
